Tool 的 Anatomy

OpenCode 能呼叫的每個 tool 都由 source code 裡的 TypeScript interface 定義。以下是這份 contract 的樣子，以及它的白話意思。

CODE 
export interface Def<Parameters extends z.ZodType = z.ZodType,
     M extends Metadata = Metadata> {
  id: string
  description: string
  parameters: Parameters
  execute(args: z.infer<Parameters>,
          ctx: Context): Effect.Effect<ExecuteResult<M>>
  formatValidationError?(error: z.ZodError): string
}
白話說明

id: string ：這個 tool 的 unique name，例如 "bash", "read", or "webfetch"。 LLM 會用這個 ID 呼叫正確 tool。

description: string：用 plain English 解釋，讓 AI 知道何時該使用這個 tool。這就是 AI 判斷時實際讀到的文字。

parameters: Parameters：這個 tool 接受的 inputs，例如 bash 的 "command" 和 "timeout"。執行前會先用 Zod validate。

execute(args, ctx) ：AI 呼叫這個 tool 時實際執行的 function。它會回傳 Effect.Effect 也就是 OpenCode 安全處理 async operations 的方式。

formatValidationError?：optional。如果 AI 傳入錯誤 inputs，可以回傳 custom error message。? 代表這個 field 不是必填。

💡
為什麼這個設計重要

每個 tool 都是 AI 和你的電腦之間的一份 contract。AI 會讀 description 來知道何時使用它；parameters 則定義如何使用它。 這種分離讓 AI agents 更安全、可預測：如果 schema 拒絕 invalid inputs，AI 就無法錯誤呼叫 tool。

📄
在 source 裡去哪裡找

Source file: packages/opencode/src/tool/tool.ts. 每個 built-in tool，包括 bash、read、edit、grep、 webfetch ，都實作同一個 interface。

魔法背後的大腦

AI agents 如何做決策，以及為什麼有三種 agent

OpenCode 不是只有一個 AI， 而是一整個團隊，每個 agent 都有不同 personality 和不同 trust level。 這就是它重要的原因。

multi-agent system 存在，是因為不同 tasks 需要不同的 risk tolerances。 不能意外刪檔的 planning AI 較適合安全探索。 能執行任何 command 的 building AI 更強大，但也需要更高信任。

不是一個 AI，而是一個團隊

每次使用 OpenCode，你其實是在和三種 specialized agents 之一合作；它們各自有不同 capabilities 和 constraints。

🏗️

Build Agent

Full permissions：可以讀檔、寫 code、執行 bash commands。 這是你的 hands-on contractor。 適用於大多數 tasks。

📋

Plan Agent

Read-only mode 可以探索與分析，但不能改任何東西。 這是安全的 research assistant。 非常適合理解 codebase。

🔍

General Agent

會在 complex searches 時自動以 subagent 形式啟動。 它在 background 蒐集資訊給 main agent。

🎯
實務經驗法則

你使用的 agent 會影響 OpenCode 被允許做什麼。 不確定要問什麼時，先用 Plan mode。 準備好要修改時，再切到 Build。

Reasoning Loop

LLM 不是只回答一次。它會在 loop 裡思考：呼叫 tools、讀取 results， 並在給 final answer 前判斷是否需要更多資訊。

1
Receive context

你的 message + conversation history + 可用 tools 會一次載入 AI。

2
Decide

哪個 tool（如果有）能幫我回答？ AI 會權衡選項，挑出最有用的下一步 action。

3
Act

呼叫 tool，然後等待 result。 這是真正的 tool call： AI 會暫停，runtime 則執行它。

4
Observe

tool 回傳了什麼？ 這會改變我的 plan 嗎？ AI 會讀 output 並更新理解。

5
Loop or Finish

如果還需要更多資訊，就回到步驟 2；如果 task 已完成，就寫出 final answer。

💡
工程師稱它為「ReAct Loop」

ReAct loop 代表 Reasoning + Acting。 它是每個 AI agent 背後的 core pattern，不只是 OpenCode。 當你看到 AI 在回答前呼叫多個 tools，你看到的正是這個 loop。

Context Window 問題

每個 model 都有一個記憶上限，稱為 context window。 它一次只能容納有限文字。

🗒️
Whiteboard Metaphor

想像一塊只能貼 200 張便利貼的 whiteboard。 當它被貼滿，你就必須擦掉最舊的便利貼，才能加入新的， 不管舊便利貼有多重要。 context window 的運作方式正是如此。

以下是真實 code，用來描述每次 response 時送給 AI 的所有東西。 它來自 packages/opencode/src/session/llm.ts：

CODE — StreamInput type 
export type StreamInput = {
  user: MessageV2.User
  sessionID: string
  parentSessionID?: string
  model: Provider.Model
  agent: Agent.Info
  permission?: Permission.Ruleset
  system: string[]
  messages: ModelMessage[]
  small?: boolean
  tools: Record<string, Tool>
  retries?: number
  toolChoice?: "auto" | "required" | "none"
}
白話說明

user ：你剛送出的 message，也就是最新 request

sessionID ：這屬於哪個 conversation

model ：要使用哪個 AI brain（Claude、GPT-4、Gemini 等）

agent ：目前啟用哪個 agent mode（build、plan 或 general）

permission ：這個 agent 被允許做什麼的 safety rules

system：定義 AI 應如何行為的 instructions（也就是 system prompt）

messages — THE KEY FIELD: 送給 AI 的 conversation history，這就是填滿 context window 的東西

tools ：AI 在這個 turn 可呼叫的完整 tools list

toolChoice ：AI 是必須使用 tool、可以自行選擇，還是完全禁止使用 tools

🗜️
Compaction：OpenCode 的 Memory Management

當 session 變得很長時，OpenCode 會執行 compaction： 它會把 conversation 較舊部分 summarized 成較短 summary，然後繼續。 你會失去一些細節，但換得更多 messages 空間。

每個 agent 都由自己的 configuration object 定義。 它在 packages/opencode/src/agent/agent.ts 裡長這樣：

CODE — Agent.Info schema 
export const Info = z
  .object({
    name: z.string(),
    description: z.string().optional(),
    mode: z.enum(["subagent", "primary", "all"]),
    native: z.boolean().optional(),
    permission: Permission.Ruleset.zod,
    model: z.object({
      modelID: ModelID.zod,
      providerID: ProviderID.zod,
    }).optional(),
    prompt: z.string().optional(),
    steps: z.number().int().positive().optional(),
  })
白話說明

name ：agent name，例如 "build"、"plan"，或你自訂的名稱

mode ：這個 agent 如何被使用：main agent、background helper（subagent），或兩者皆是

permission：這個 agent 必須遵守的 permission ruleset，也就是核心 safety constraint。

model：optional。可為這個特定 agent override 使用哪個 AI model，不受 global setting 影響。

prompt：optional custom system prompt，用來改變這個 agent 的 personality 或 instructions。

steps ：停止前最多 tool calls 數量，用來防止 runaway loops 的 safety limit

🔑
為什麼用 Zod？

z.object() syntax 來自 Zod schema， 也就是給 TypeScript 用的 validation library。 這代表 OpenCode 會在 runtime validate agent configurations；如果你定義 custom agent 時填了 bad values，它會立刻以清楚 error 失敗，而不是靜默出錯。

Tool 的 Anatomy：Bash 範例

每個 tool 都由一個 schema — a precise description of what inputs it accepts. Here is the real schema for the bash tool from OpenCode source code, alongside a plain English translation.

CODE 

const Parameters = z.object({
  command: z.string()
    .describe("The command to execute"),
  timeout: z.number()
    .describe("Optional timeout in milliseconds")
    .optional(),
  workdir: z.string()
    .describe("The working directory to run
     the command in...")
    .optional(),
  description: z.string()
    .describe("Clear, concise description
     of what this command does..."),
})
PLAIN ENGLISH

command — The actual command to run — like npm test or git status

timeout (optional) — A time limit in milliseconds — stop the command if it takes too long

workdir (optional) — Which folder to run the command in — like running npm install inside a specific package folder

description — A plain English description of what this command does — helps you review what the AI is about to run

💡
Why description is required — not optional

這會強制 AI 在每次 bash command. You can always see WHY it wants to run something, not just WHAT it is running.

03

Permission System：你的 Safety Net

permission system 讓你能細緻控制每個 tool call。有三種 actions 可用：

✅
allow

Run without asking — you have pre-approved this action

⏸️
ask

Pause and show a confirmation prompt — you decide in the moment

🚫
deny

Block completely — this action is never allowed

Example rules in JSON configuration

bash → * → ask Must confirm all bash commands before they run — the wildcard catches everything
bash → npm test → allow Running tests is pre-approved — no confirmation needed for this specific command
edit → *.env* → ask Ask before editing any .env file — the glob pattern *.env* matches any env file
edit → src/** → allow Free to edit anything in the src/ folder — the AI works uninterrupted on source code
bash → rm -rf → deny Block this dangerous command entirely — it can never run, no matter what
📋
最 specific 的 rule 優先

Rules are checked in order — the most specific match takes precedence. A rule for npm test is more specific than a rule for *, so it overrides the wildcard.

What is MCP? Model Context Protocol 是把 external tools 接到 AI 的 industry standard。就像 USB port：任何依 spec 建好的 tool，都能插進支援它的 AI。

MCP servers 可以包裝 internal tools、APIs、databases 和 services，並透過同一套 tool interface as built-in capabilities.

02

Session Table：Code Decoded

你開啟的每個 session 都會成為一列，存在 session table of the schema. Here is the real code from OpenCode, translated into plain English.

CODE 

export const SessionTable = sqliteTable(
  "session",
  {
    id: text().primaryKey(),
    project_id: text()
      .references(() => ProjectTable.id),
    workspace_id: text(),
    parent_id: text(),
    slug: text().notNull(),
    directory: text().notNull(),
    title: text().notNull(),
    share_url: text(),
    summary_additions: integer(),
    summary_deletions: integer(),
    summary_files: integer(),
    summary_diffs: text({ mode: "json" }),
    revert: text({ mode: "json" }),
    permission: text({ mode: "json" }),
  }
)
PLAIN ENGLISH

id ：這個 session 的 unique ID，就像收據號碼。這是 primary key.

project_id ：這個 session 屬於哪個 project folder。

workspace_id — Optional: which enterprise workspace this belongs to.

parent_id ：如果你從另一個 session fork，這會指向原 session。

slug — A short URL-friendly name for the session.

directory ：OpenCode 執行所在的電腦 folder。

title ：這個 session 的 display name（顯示在 UI 裡）。

share_url ：如果你 share 這個 session，加密 link 會存在這裡。

summary_additions / deletions / files ：新增幾行、刪除幾行、修改幾個 files。

summary_diffs — A JSON summary of every file that was changed.

revert — The checkpoint data needed to undo everything back to this point.

permission ：這個 session 期間啟用的 permission rules。

💡
30 秒理解 Relational Databases

Storing data in structured tables like this is called a relational database. Think of it like a spreadsheet where each row is one session and each column is a piece of information. SQLite is a tiny, fast database that lives as a single file on your computer — no server needed.

OpenCode uses Drizzle ORM to define these tables in TypeScript, making the migrations type-safe and keeping the blob fields strongly typed.

03

Configuration Hierarchy

OpenCode reads settings from three layers, each overriding the one below it. Understanding this hierarchy tells you exactly where to put any setting.

1
System / Enterprise (MDM)

Managed by IT via Apple MDM or Windows GPO. Cannot be overridden by users. Used in enterprise deployments to enforce company-wide settings like approved AI providers.

2
User-level (~/.opencode/config.jsonc)

跨所有 projects 的個人 defaults。你會在這裡設定偏好的 AI model、default permissions 和個人 MCP servers.

3
Project-level (.opencode/config.jsonc)

project-specific settings，會 override 你的個人 defaults。你會在這裡設定這個 codebase 最適合的 model、project-specific permissions，以及 team-shared MCP servers。

檔案放在哪裡

~/ your home folder
.opencode/ personal OpenCode folder
config.jsonc your personal settings — applies to all projects
your-project/ any project folder
.opencode/ project-level OpenCode folder
config.jsonc project settings — overrides personal settings

重要 config options

model Which AI to use: claude-sonnet-4-5, gpt-4o, gemini-1.5-pro, etc.
provider Which AI provider: anthropic, openai, google, local, etc.
permission Your allow / deny / ask rules for tools and file paths.
mcp External tool servers to connect at startup.
plugins npm packages that add new tools or UI features.
📄
為什麼用 .jsonc 而不是 .json？

JSONC is JSON with Comments — the c stands for comments. It lets you add notes like // my work laptop API key right inside the config file, which regular JSON does not allow.

04

Revert、Fork、Share

Because every change is recorded, OpenCode can give you three superpowers that would be impossible without the database.

1
Revert

Go back to any message checkpoint. All file changes made after that point are undone. Like time travel for your codebase. Use this when OpenCode went in the wrong direction.

2
Fork

Create a new session that branches off at a specific point. Try approach A in one branch, approach B in another. Each branch has its own history and can be reverted independently.

3
Share

Generate an encrypted link to this session. Anyone with the link can see exactly what you asked and what changed. Useful for reviewing AI work with a colleague or reporting a bug.

⏪
Revert 實際如何運作

Revert does not just undo the AI messages — it restores the actual files on disk to the state they were in at that checkpoint。 file diff stored in the database is replayed in reverse.

🌿
Fork vs. Git Branch

A session fork is different from a git branch. Forks live inside OpenCode's own history — they track the conversation and all AI actions, not just the file state. You can have a fork per idea, then revert the ones that did not work, all without touching your git history.

02

Real-Time Updates 如何運作

WebSocket is like a phone call — once you dial, the line stays open and either side can speak at any moment. HTTP requests are like letters — you send one, wait for a reply, then the connection closes.

當 AI 工作時，每個 event：file edit、tool call、message，都會透過 WebSocket broadcast 給每個已連線的 client simultaneously.

📡

03

Event Bus：Code Decoded

event bus 定義在 packages/opencode/src/bus/bus.ts。以下是真實 interface，以及每個 method 的白話說明。

CODE — Bus.Interface 
export interface Interface {
  readonly publish: <D extends BusEvent.Definition>(
    def: D,
    properties: z.output<D["properties"]>,
  ) => Effect.Effect<void>
  readonly subscribe: <D extends BusEvent.Definition>(def: D) => Stream.Stream<Payload<D>>
  readonly subscribeAll: () => Stream.Stream<Payload>
  readonly subscribeCallback: <D extends BusEvent.Definition>(
    def: D,
    callback: (event: Payload<D>) => unknown,
  ) => Effect.Effect<() => void>
  readonly subscribeAllCallback: (callback: (event: any) => unknown) => Effect.Effect<() => void>
}
PLAIN ENGLISH

publish(def, properties) — Send an event to all listeners — like a radio tower broadcasting a signal

subscribe(def) — Listen for one specific type of event — like tuning to a specific radio frequency

subscribeAll() — Listen for ALL events — useful for debugging or logging everything that happens

subscribeCallback(def, callback) — Listen for a specific event and run a function when it arrives

subscribeAllCallback(callback) — Run a function every time ANY event fires — the most powerful listener

🔗
保持同步的秘密

This interface is what lets the terminal UI, web UI, and VS Code extension all stay in sync without talking to each other directly。y all listen to the same bus。 bus carries the truth.

04

在 OpenCode 之上建構

Because OpenCode exposes a standard REST API and WebSocket, you can build anything on top of it. Here are three real-world use cases.

💬
Use Case 1: Slack Bot

Send a message to Slack with a coding task → your bot calls POST /sessions to start a session → POST /sessions/{id}/messages to send the task → WebSocket stream returns the result → bot replies in Slack with the outcome.

🔄
Use Case 2: CI/CD在tegration

After every pull request → your pipeline starts an OpenCode session → asks it to review the diff for security issues → gets a structured response → posts it as a PR comment.

🎨
Use Case 3: Custom Dashboard

Build any frontend you want — React, Vue, plain HTML — that connects to the REST API. Your own branding, your own UI, same powerful engine underneath. It is headless by design.

4 個關鍵 API endpoints

POST /sessions Create a new session in a project directory — this is where every interaction begins
POST /sessions/{id}/messages Send a message and start the AI working — the core interaction endpoint
GET /sessions/{id}/messages Retrieve the full conversation history — useful for displaying past sessions or building summaries
WS /sessions/{id}/subscribe Open a WebSocket to receive live events as they happen — the real-time stream of everything the AI is doing

Create a .opencode/config.jsonc in your project and set permission rules. Start with bash set to ask — see every command before it runs. You are always in control.
