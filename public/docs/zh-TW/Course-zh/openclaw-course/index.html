02

五個角色

每當你送出一則 message，OpenClaw 裡五個不同部件就會開始行動。來認識它們。

你機器裡的機場

把 OpenClaw 想成一座繁忙機場。航廈是 Gateway；航空公司是 Channels；航管是 Router；駕駛艙組員是 Pi Agent；地勤服務則是 Plugins。

🏛️
Gateway

WebSocket control plane，跑在你機器上的 ws://127.0.0.1:18789。 每個 channel、CLI command 和 app 都透過這裡連線。

📡
Channels

20 多個 adapters，負責在 OpenClaw 與 WhatsApp、Telegram、Slack、Discord、Signal、iMessage 等平台之間轉換。 每個 channel 都會說該 app 的語言：webhooks、polling，全都包含。

🗺️
Router

讀取每則 incoming message，並決定：哪個 AI agent 處理它？ 它屬於哪個 conversation thread？Router 會產生一個 session key。

🧠
Pi Agent

AI runner。它保存 conversation history、呼叫 Claude/GPT/Gemini、執行 tools，並把 reply 經由 Gateway stream 回去。

🔌
Plugins

可加入新 channels、新 AI providers 或新 tools 的 optional npm packages。 Core 保持精簡；optional capabilities 則以 plugins 形式發佈。

✈️
為什麼是五個？

每個角色都只有一個明確職責。當事情出錯時，你會知道該盤問哪個角色：找那個職責和症狀相符的角色。

認識 Router

Router 的工作是回答一個問題：「這則 message 應該由哪個 AI agent 和 conversation thread 處理？」 它會產生 ResolvedAgentRoute，也就是一張 decision slip，裡面有 Gateway dispatch 工作所需的一切資訊。

CODE — src/routing/resolve-route.ts (lines 40–61) 
export type ResolvedAgentRoute = {
  agentId: string;
  channel: string;
  accountId: string;
  sessionKey: string;
  mainSessionKey: string;
  lastRoutePolicy: "main" | "session";
  matchedBy:
    | "binding.peer"
    | "binding.peer.parent"
    | "binding.peer.wildcard"
    | "binding.guild+roles"
    | "binding.guild"
    | "binding.team"
    | "binding.account"
    | "binding.channel"
    | "default";
};
白話說明

agentId：由哪個 AI agent 處理這則 message。

channel：message 來自哪個 app（telegram、whatsapp、discord……）。

accountId：是哪個 bot account 收到它（你可能有多個 bot accounts）。

sessionKey：這個 conversation thread 的 unique ID。

mainSessionKey：direct（非群組）chats 的 shortcut alias。

lastRoutePolicy：要更新 main session 還是特定 session。

matchedBy：router 是如何找到正確 agent 的；這是 debugging 時最有用的欄位。

"binding.peer" 表示精確 user-ID match，是最 specific 的 binding rule。

"default" 表示沒有任何 rules match，因此使用 fallback agent。

可以把它想成每則 message 上的 shipping label。

🔍
Debugging 小提醒

如果 messages 被送到錯誤 agent，請檢查 logs 裡的 matchedBy field。 如果值是 "default"，代表沒有 binding rule match，請檢查 binding configuration。

為什麼回覆會一個字一個字出現

While the AI is generating a reply, OpenClaw sends partial text to Telegram so you see it appear in real time. 這就是 Draft Stream Loop，也就是 throttled loop that flushes pending text at a controlled rate.

CODE — src/channels/draft-stream-loop.ts (lines 10–50) 
export function createDraftStreamLoop(params: {
  throttleMs: number;
  isStopped: () => boolean;
  sendOrEditStreamMessage: (text: string) => Promise<void | boolean>;
}): DraftStreamLoop {
  let lastSentAt = 0;
  let pendingText = "";
  let inFlightPromise: Promise<void | boolean> | undefined;

  const flush = async () => {
    while (!params.isStopped()) {
      if (inFlightPromise) {
        await inFlightPromise;
        continue;
      }
      const text = pendingText;
      if (!text.trim()) { pendingText = ""; return; }
      pendingText = "";
      const current = params.sendOrEditStreamMessage(text).finally(() => {
        if (inFlightPromise === current) inFlightPromise = undefined;
      });
      inFlightPromise = current;
      const sent = await current;
      if (sent === false) { pendingText = text; return; }
      lastSentAt = Date.now();
      if (!pendingText) return;
    }
  };
PLAIN ENGLISH

Creates a streaming loop that sends text to the channel as the AI generates it

throttleMs = how often to send updates — prevents flooding Telegram every millisecond

isStopped() = check if the run has been cancelled before sending more

sendOrEditStreamMessage = 實際用新文字編輯 Telegram message 的 function

—

lastSentAt = timestamp of last successful send, used for throttle timing

pendingText = AI text waiting to be sent to Telegram

inFlightPromise = a Telegram edit is already in progress — wait for it before sending more

—

flush() = the core loop: grab all pending text, send it, wait, check again

If isStopped() is true — stop immediately, the run was cancelled

如果 Telegram request 還在進行中，就等它完成再進入下一輪 loop

Grab all pending text, then clear the buffer so new AI text can accumulate

Send the text to Telegram and track the request as “in flight”

If sent === false: Telegram rejected this batch — hold onto the text and retry next cycle

Update lastSentAt, then check if more text arrived while we were waiting

⚡
為什麼要 throttle？

Telegram has a rate limit 對每秒能編輯 message 幾次有限制。 throttleMs parameter ensures OpenClaw never hits that limit, even during a fast-streaming response.

每則 message 上的 routing label

Router 處理 message 時，會建立一個 session key — a unique ID that identifies this exact conversation. 同一段 conversation 裡的每則 message 都會拿到相同 key，因此 Pi Agent 知道要載入哪份 history。

🤖
agentId

哪個 AI agent handles this thread (e.g. "default")

📡
channel

哪個 app（例如 "telegram", "whatsapp")

🏦
accountId

哪個 bot account 收到 message（例如 "bot1")

👤
peer

Who sent it — user ID or group ID (e.g. "user_123456")

🔑
Example key: default/telegram/bot1/user_123456

Session keys are why OpenClaw can run hundreds of separate conversations simultaneously — each gets its own key, its own file on disk, and its own AI context. 當你在 Telegram 問 AI 關於 project A、在 Discord 問 project B，這兩段會是完全分開的 conversations。

「Do Not Disturb」系統

While a reply is being generated, OpenClaw tracks whether each conversation is busy. 這可以避免第二則 message 打斷仍在寫作中的 reply。

CODE — src/channels/run-state-machine.ts (lines 18–34) 
export function createRunStateMachine(params: RunStateMachineParams) {
  const heartbeatMs = params.heartbeatMs ?? DEFAULT_RUN_ACTIVITY_HEARTBEAT_MS;
  let activeRuns = 0;
  let runActivityHeartbeat: ReturnType<typeof setInterval> | null = null;

  const publish = () => {
    params.setStatus?.( {
      activeRuns,
      busy: activeRuns > 0,
      lastRunActivityAt: now(),
    });
  };
PLAIN ENGLISH

為單一 conversation thread 建立 state tracker

heartbeatMs = send a status update this often while a run is active

activeRuns = how many AI runs are currently happening (normally 0 or 1)

runActivityHeartbeat = a repeating timer that fires status updates while busy

—

publish() = broadcasts the current state to anyone listening

activeRuns = the live count of ongoing AI calls in this thread

busy: activeRuns > 0 = true 表示 reply 正在進行中；channel 會把下一則 message 排入 queue

lastRunActivityAt = timestamp of the last activity, used for health monitoring

Like an “on air” light in a radio studio — red means someone is talking, wait your turn

🔴
Busy (activeRuns > 0)

reply 正在生成。新的 messages 會先排入 queue，等目前 run 完成後再送出。

🟢
Idle (activeRuns === 0)

沒有 reply 正在進行。下一則 message 會觸發新的 inference loop immediately.

💓
Heartbeat

While busy, a periodic pulse is sent so the Gateway knows the run is still alive — not silently crashed.

04

內部大腦

Pi Agent 保存 conversation history，決定要呼叫哪個 AI model，執行 browser 或 bash terminal 這類 tools，並把 answer stream 回去。以下是它的思考方式。

你的 conversation 存在 disk 上

A session is a list of messages — yours and the AI’s — stored as a JSONL file on your disk. 每則新 message 都會 append 到那個檔案；每次 AI call 前，整份 history 都會被打包並作為 context.

📜
Session History

完整 conversation 會以 JSONL file 保存。AI 在生成每次 reply 前，都會讀到先前每則 message：包含你的，也包含它自己的。

🔑
Session Key

這段 conversation 的 unique ID（來自 Router）。Pi Agent 用它在 disk 上找到正確檔案。

✂️
Compaction

當 history 大到 AI 無法處理時，最舊的 messages 會被 summarized 成精簡 synopsis，就像把逐字稿濃縮成會議筆記。

🗂️
Parallel Sessions

Each channel + peer combination gets its own session. 你 can have hundreds of separate AI conversations running simultaneously — they never interfere.

一個 AI 失敗，就試下一個

OpenClaw supports Claude, GPT, Gemini, Llama, and dozens of other models. 如果 provider 被 rate-limited 或掛掉，它會自動嘗試下一個 configured model，這個機制稱為 model failover。

CODE — src/agents/model-fallback.ts (lines 45–60) 
export class FallbackSummaryError extends Error {
  readonly attempts: FallbackAttempt[];
  readonly soonestCooldownExpiry: number | null;

  constructor(
    message: string,
    attempts: FallbackAttempt[],
    soonestCooldownExpiry: number | null,
    cause?: Error,
  ) {
    super(message, { cause });
    this.name = "FallbackSummaryError";
    this.attempts = attempts;
    this.soonestCooldownExpiry = soonestCooldownExpiry;
  }
}
PLAIN ENGLISH

FallbackSummaryError = the error thrown only when every configured model has been tried and failed

attempts = a log of each model tried and why it failed (rate limit? timeout? auth error?)

soonestCooldownExpiry = the earliest time any model will accept requests again

—

message = a human-readable summary of all the failures

attempts = the full list of what was tried, in order

soonestCooldownExpiry = when to retry (milliseconds timestamp)

cause = the last underlying error, for debugging purposes

—

Like a rejection slip with full notes: tried Claude (rate-limited), tried GPT (timed out), tried Gemini (auth error). Try again in 47 seconds.

1️⃣
嘗試 primary model

第一個 configured provider（通常是 Claude 或 GPT）會先收到 request。

❌
Primary 失敗

Rate-limited, timed out, or auth error. Record the attempt and move to the next provider.

2️⃣
嘗試下一個 provider

Each fallback is tried in sequence until one succeeds or all are exhausted.

🚫
全部失敗

Throw FallbackSummaryError with the full attempt log and the soonest retry time.

會被寫滿的 whiteboard

每個 AI model 一次能讀的文字量都有上限，稱為 context window. Think of it like a whiteboard: you can only fit so much writing before you have to erase the oldest notes.

“Context window” is one of the most important concepts in AI. It explains why your AI sometimes “forgets” something from earlier in a long conversation — the old messages fell off the whiteboard. 當你要求 AI agent「improve the bot’s memory」時，你其實是在要求它處理 context window more cleverly.

📏
Token budget

Each model has a fixed context window — typically 128k–200k tokens for modern models. Longer conversations eat into this budget faster.

🗜️
觸發 compaction

當 history 接近上限時， compaction summarizes the oldest messages into a brief synopsis and replaces them.

📝
哪些內容會被保留

synopsis 會保留早期 conversation 的大意。具體細節（名字、數字、精確指令）可能遺失，這是已知 tradeoff。

Skills、Tools 與 Plugin Machine

OpenClaw 內建 browser、bash terminal、canvas 和 cron scheduler。它也有一套 plugin system that lets you bolt on new channels, new AI providers, and entirely new capabilities. Here is how the whole extension machine is wired together.

擴充 OpenClaw 的三種方式

Think of OpenClaw like a Swiss Army knife factory. The knife ships with built-in blades — and you can order add-on modules from the store that clip onto the handle. These three extension layers give you complete control over what your AI can do and how it behaves.

🔧
Tools

Capabilities the AI can use right now: browser control, bash terminal, canvas drawing, cron scheduling, memory search, camera. They live in the Pi Agent and are called when the AI decides it needs them.

📦
Plugins

npm packages (or local workspace folders) that register new channels, new AI model providers, or new tools. Loaded by the Gateway at startup. Core stays lean — everything optional is a plugin.

📋
Skills

Markdown files with specialized instructions that get injected into the AI's system prompt. Install a release-management skill and the AI knows your release process by heart.

💡
為什麼需要三層？

這種三層設計是刻意的。Core 只做最少必要工作；其他全部 pluggable。這代表你可以新增 messaging app，而不必改 OpenClaw core 的任何一行。

AI 的 toolbox

AI 能使用的每個 tool 都會註冊在 tools catalog，這是一張把 tool names map 到 handler functions 的 lookup table。 當 AI 說它想使用 browser tool 時，Pi Agent 會用名稱找到它並呼叫。

CODE — Tool Catalog Pattern 
export type ToolEntry = {
  name: string;
  description: string;
  inputSchema: unknown;
  handler: ToolHandler;
};

export type ToolCatalog = {
  tools: ToolEntry[];
  findTool(name: string): ToolEntry | undefined;
};
PLAIN ENGLISH

ToolEntry = one capability in the catalog

name = AI 呼叫這個 tool 時使用的名稱（例如 bash, browser_snapshot, canvas_eval)

description = what this tool does — the AI reads this to decide when to use it

inputSchema = the shape of inputs this tool expects (e.g. bash needs a command string)

handler = the actual code that runs when the AI calls this tool

—

ToolCatalog = the master list of all tools

findTool(name) = look up a tool by name — like finding a recipe by dish name

When the AI says “I want to run bash df -h”, the Pi Agent calls findTool('bash') and passes the command

plugins 如何自我介紹

每個 plugin 都會透過 manifest file 宣告自己，也就是一份說明「我是誰、我提供什麼」的 JSON file。 Gateway 會在 startup 時讀取所有 manifests，並在尚未載入任何 plugin code 前，建立自己的 plugin registry。

openclaw.plugin.json — the plugin identity card
"id" Unique plugin identifier — e.g. "matrix", "zalo-user", "anthropic-vertex"
"openclaw.channel.id" If it is a channel plugin, the channel identifier used in routing config
"openclaw.install.npmSpec" The npm package name used for installation — e.g. "@openclaw/matrix"
"entrypoint" The JavaScript file to execute when the plugin loads — only read after the manifest is approved

給 AI 的護貝 instruction cards

Skills are Markdown files that inject specialized instructions into the AI system prompt. Install a code-review skill and the AI knows your team’s review process. Install a release-manager skill and it handles your versioning workflow.

CODE — src/agents/skills.ts 
export function resolveSkillsInstallPreferences(
  config?: OpenClawConfig
): SkillsInstallPreferences {
  const raw = config?.skills?.install;
  const preferBrew = raw?.preferBrew ?? true;
  const manager = normalizeLowercaseStringOrEmpty(
    normalizeOptionalString(raw?.nodeManager)
  );
  const nodeManager: SkillsInstallPreferences["nodeManager"] =
    manager === "pnpm" || manager === "yarn" ||
    manager === "bun" || manager === "npm"
      ? manager
      : "npm";
  return { preferBrew, nodeManager };
}
PLAIN ENGLISH

Reads your config to figure out how to install skills

config?.skills?.install = safely read the install preferences (the ?. means: only if it exists)

preferBrew = prefer Homebrew for installs on Mac (defaults to true)

—

nodeManager = which JavaScript package manager to use for installs

Accepts: pnpm, yarn, bun, or npm

Anything else falls back to npm

—

Returns a clean preferences object used by openclaw skills install

This is the configuration resolver — it translates your config file into decisions

always-on heartbeat checker

每個 channel 都有 health policy，定義要多積極地監控 connection。 如果 channel 沉默太久，monitor 就會觸發 reconnect。

CODE — Channel Health Policy 
export type ChannelHealthPolicy = {
  enabled: boolean;
  probeIntervalMs: number;
  staleThresholdMs: number;
  maxConsecutiveFailures: number;
};

export const DEFAULT_CHANNEL_HEALTH_POLICY: ChannelHealthPolicy = {
  enabled: true,
  probeIntervalMs: 60_000,
  staleThresholdMs: 300_000,
  maxConsecutiveFailures: 3,
};
PLAIN ENGLISH

ChannelHealthPolicy = the rules for monitoring one channel’s connection

enabled = whether to monitor at all (yes, by default)

probeIntervalMs = check every 60 seconds (60,000 ms = 1 minute)

staleThresholdMs = if no activity for 5 minutes (300,000 ms), consider the channel stale

maxConsecutiveFailures = declare the channel unhealthy after 3 failed checks in a row

—

DEFAULT_CHANNEL_HEALTH_POLICY = the standard settings applied to every channel

Like a smoke detector: checks regularly (every 60s) and raises an alarm if 5 minutes pass without a heartbeat
