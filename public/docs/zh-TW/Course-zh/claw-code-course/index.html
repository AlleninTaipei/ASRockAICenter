Claw 如何運作
01

Claw 會做什麼

一個住在 terminal 裡，而且真的能動手做事的 AI

想像你有一個永不休息的 coding assistant

你輸入一句話。它會讀你的檔案、執行指令、編輯程式碼、搜尋網路，然後回報它做了什麼。這就是 Claw Code：一個 open-source CLI，把 AI coding agent 放進你的 terminal。

它不是聊天視窗。它是 coding agent，也就是說，它可以代表你在電腦上讀檔、修改程式碼、執行指令。

💬

Interactive REPL

進行來回對話。你可以問問題、下指令，並接續前面的 turns 繼續工作。

⚡

One-Shot Prompts

用非互動模式執行單次指令：claw prompt "summarize this repo"

🤖

Multi-Provider AI

可搭配 Claude、GPT、Grok、Qwen，甚至你自己機器上跑的 local models。

想像你輸入：claw prompt "say hello"

接下來一秒內，從鍵盤到 AI、再回到你眼前，會發生這些事。

1
CLI 啟動

claw binary 會啟動、載入你的 config、檢查 API key，並組出一段「system prompt」，告訴 AI 目前有哪些 tools 可以用。

2
你的訊息被包裝起來

你的文字（"say hello"）會被包成一個帶有 role（"user"）的結構化 message object，然後透過 HTTPS 送給 AI。

3
AI 以 stream 方式回應

你不用等完整答案產生完；文字會像即時打字動畫一樣一個個出現。AI 生成多少，就 stream 回來多少。

4
渲染到你的 terminal

回應會被格式化：markdown 轉成可讀文字、code blocks 加上 highlight，最後印到你的畫面上。

5
Session 被保存

整段對話會寫入 JSONL 檔，讓你之後可以用 --resume latest 接著做。

兩種模式並排看

從 CLI source code 來看，它如何判斷你要的是快速 one-shot 回答，還是完整 interactive session：

版本號會在 compile time 被烤進 binary，就像出廠時印上的序號。

精確的 git commit 也是一樣，讓你永遠知道自己跑的是哪個 build。

這是 optional：只有 build system 有設定時才會出現。

如果 tool 已經跑完，但 AI 後面沒有回應，先等 10 秒再判定可能出問題。

💡
為什麼用 Rust？

Claw Code 是用 Rust 寫的。Rust 可以 compile 成單一高速 binary，沒有額外 runtime overhead。這代表啟動更快、較少 memory bug 造成的隱性 crash，而且只要一個檔案就能到處發佈。當你執行 cargo build，就會得到一個不需要額外安裝東西的 claw executable。

最重要的幾個指令

建好之後，這幾個 entry points 會是你每天最常用的：

claw 啟動 interactive REPL：輸入訊息、按 Enter、看回應。就像 terminal 裡的 chat app。
claw prompt "…" One-shot mode：傳入單次指令，完成後就結束。很適合 scripts 和 automation。
claw --resume latest 從上次離開的地方繼續：所有先前 context 都會從保存的 session file 還原。
/doctor 內建 health check。安裝後先跑它，確認 API key、permissions 和 configuration 都設定正確。
🔑
一個前提：API key

Claw 會透過網路連到 AI models，因此需要 API key。執行 claw 前，你需要在 terminal 裡設定 export ANTHROPIC_API_KEY="sk-ant-..."。只有 Claude subscription 不夠，還需要從 console.anthropic.com 取得獨立的 API key。

02

登場角色

八個 Rust crates，各自負責不同工作。認識每個 claw session 背後的團隊。

Claw 不是一個程式，而是八個

可以把它想成一齣舞台劇。觀眾看到的是一場表演，但後台有演員、燈光、音控、導演、舞台監督，各自扮演特定角色。Claw 也是如此：八個 crates 一起工作，每個 crate 負責一件事。

rust/crates/ Rust workspace：八個 crates 都在這裡
rusty-claude-cli/ 🎭 導演：你真正執行的 main binary
runtime/ 🧠 大腦：conversation loop、sessions、hooks、MCP
api/ 📡 無線電：透過網路和 Anthropic、OpenAI、xAI 溝通
tools/ 🔧 雙手：執行 file reads、bash commands、searches
plugins/ 🔌 擴充板：載入 custom hooks 和 plugin scripts
commands/ 📋 手冊：處理 /help、/status 這類 slash commands
telemetry/ 📊 帳本：追蹤 token usage 和 cost
mock-anthropic-service/ 🧪 排練舞台：測試用的 fake API server

負責大部分工作的三個 crates

🎭
rusty-claude-cli — 前門

這就是 claw binary。你看得到的東西幾乎都由它處理：解析 command-line flags、在 AI 思考時畫出 spinner、把收到的文字 stream 到 terminal，並把其他 crates 串起來。它是唯一知道你螢幕長什麼樣子的 crate。

🧠
runtime — Conversation Brain

這個 crate 掌管每個 turn 的邏輯：建立 message payload、決定 AI 可以使用哪些 tools、執行 permissions、把 session 存到 disk、在 tool calls 前後執行 hooks，並壓縮太長的對話，讓它塞得進 AI 的 memory window。

📡
api — Network Layer

這個 crate 負責和 AI providers 溝通。它知道 Anthropic API 和 OpenAI API 的 request 格式差異、如何解析 streaming response，以及如何實作 prompt caching（重複 requests 時可節省成本的技巧）。

api crate 如何處理多個 AI providers

api crate 使用 Rust 裡稱為 enum 的 pattern，表示目前啟用的是哪個 AI provider。這是真實程式碼：

CODE — api/src/client.rs 
pub enum ProviderClient {
  Anthropic(AnthropicClient),
  Xai(OpenAiCompatClient),
  OpenAi(OpenAiCompatClient),
}

pub fn from_model(model: &str)
  -> Result<Self, ApiError> {
  let resolved = resolve_model_alias(model);
  match detect_provider_kind(&resolved) {
    Anthropic => Ok(Self::Anthropic(...)),
    Xai => Ok(Self::Xai(...)),
    OpenAi => Ok(Self::OpenAi(...)),
  }
}
白話說明

ProviderClient 永遠只會是三種之一：Anthropic client、xAI (Grok) client，或 OpenAI-compatible client。

Anthropic 有自己的 client type，因為它的 API 和 OpenAI 不同。

xAI (Grok) 使用和 OpenAI 相同的 wire format，所以可以共用 implementation。

OpenAI 也是同一套做法：一個 implementation 可以服務多個 providers。

 

給定像 "gpt-4o" 或 "claude-opus-4-6" 這樣的 model name，先判斷要使用哪個 provider……

……先把 "opus" 這類 shorthand 展開成 "claude-opus-4-6"……

……再看名稱決定："claude" → Anthropic，"grok" → xAI，其他 → OpenAI-compat。

💡
這稱為 Strategy Pattern

不管底層啟用的是哪個 provider，同一個 function call（stream(request)）都能運作。enum 把差異藏起來。這是軟體裡最常見的 design patterns 之一：讓不同 implementations 共享同一個「interface」，這樣其他程式碼就不必知道自己正在用哪一個。

一次 Turn 的內部流程

從你按下鍵盤到 AI 回應，每一步都用動畫拆開看

對話是 loop，不是一條線

多數人會把 AI 想成一通電話：你說話、它聽、它回話，結束。但 coding agent 更像一場 郵件接力賽：你的訊息是一封信，交給第一位跑者，再轉交給下一位；在回信送回來之前，跑者可能還得繞路去拿資料。

每個「turn」都可能包含 多次 round trips，因為 AI 可能需要先呼叫 tools（讀檔、執行指令），才能給你最終答案。完整圖像如下：

為什麼回應會一個字一個字出現

當你問 AI 一個問題時，它不是先生成完整答案再一次送出。它會一次生成一個 token，而 Claw 會在每個 token 抵達的瞬間顯示出來。

🎬
把它想成即時體育比分 ticker

體育比分 ticker 不會等比賽結束才報分數，而是邊發生邊 stream 即時更新。SSE (Server-Sent Events) 也是同樣概念：connection 維持開啟，資料持續流過來。Claw 使用 SSE decoder 解析每個 incoming chunk，並立刻顯示。

CODE — runtime/src/conversation.rs 
pub enum AssistantEvent {
  TextDelta(String),
  ToolUse {
    id: String,
    name: String,
    input: String,
  },
  Usage(TokenUsage),
  MessageStop,
}
白話說明

AssistantEvent 代表 AI stream 裡可能抵達的四種事件之一。

TextDelta：一小段文字（可能是一個字、半個字，或只是一個空白）。立刻 render。

ToolUse：AI 想呼叫某個 tool。它包含……

……一個 unique ID（用來追蹤這是哪一次 tool call）……

……tool name（例如 "read_file"）……

……以及 arguments（例如要讀哪個 file path）。

Usage：token counts，也就是用了多少 tokens（用於 billing 和 cost tracking）。

MessageStop：AI 已經生成完畢。清理狀態，等待下一個 turn。

對話太長時會發生什麼

AI models 有一個 context window，也就是它們記憶中能容納的最大文字量。session 變長後，claw 會自動對 conversation 做「compaction」。

1
估算 token usage

runtime 會計算目前 session 使用了多少 tokens。當超過約 100,000 tokens，就會啟動 compaction。

2
請 AI 做 summary

AI 會被要求把目前為止發生的事寫成精簡 summary。只有最後 4 則 messages 會原文保留。

3
用 summary 取代 history

所有舊 messages 會被 summary 替換。session 仍保有完整脈絡，但 footprint 小很多。

💡
這對 AI director 為什麼重要

當你看到 claw 說「this session is being continued from a previous conversation」時，代表 compaction 已經觸發。如果你正在任務中途，AI 可能會失去早期決策的細節。你可以透過為大型獨立任務開新 sessions，或在指令中明確說明哪些 context 很重要，來降低這個風險。

04

Tools 與 Permission System

AI 如何安全地在你的機器上做真實工作，以及誰決定它被允許做什麼

AI 可以讀你的檔案。那它也該能刪掉嗎？

Claw 給 AI 真正的 tools：它可以讀檔、寫檔、執行 bash commands、搜尋 codebase，也能發出 network requests。這非常強大，也正因如此，AI 的 requests 和你的機器之間需要一套 permission system。

把它想成一棟有門禁卡系統的公寓。AI 是來訪的承包商：它可以進入被授權的房間，但大樓管理員（permission system）控制它的卡能開哪些門。

read-only AI 可以讀檔並搜尋 codebase，但不能寫入、刪除或執行 shell commands。適合安全探索。
workspace-write AI 可以讀寫目前 project directory 內的檔案，也能執行不影響 workspace 外部的 bash commands。
danger-full-access 沒有任何限制。AI 可以執行任何指令、修改任何檔案。只有在你完全理解並信任任務時才使用。
⚠️
permission mode 不是 AI 設定，而是硬邊界

即使你請 AI「刪掉所有東西」，在 read-only mode 下它實際上做不到，因為 permission enforcer 會在 tool call 執行前擋下來。AI 的意圖不會覆蓋 mode。

AI 實際上有哪些 tools？

tools crate 提供一組 AI 可以 request 的 built-in capabilities。這些會預先宣告在 system prompt 裡，所以 AI 在看到你的第一則訊息前，就知道自己能做什麼。

📄

read_file

讀取任意檔案內容。支援 offset 和 line limits，因此能讀大型檔案的特定片段。

✏️

write_file / edit_file

建立新檔案，或對既有檔案套用精確 diff。AI 會送出精確的 old/new strings，不靠猜。

🔎

glob_search / grep_search

用 pattern 尋找檔案（例如 **/*.rs），或用 regex 搜尋檔案內容。相當於 find 和 grep。

💻

execute_bash

執行 shell command。需要 workspace-write 或更高權限。output（stdout + stderr）會回傳給 AI。

🔌

MCP tools

MCP servers 可以暴露額外 tools，例如 database queries、web browsing、custom APIs，並沿用同一套 permission model。

tool call 執行前如何被檢查

每一次 tool invocation 都會經過 permission system。以下是真實代表 permission check 的 data structure：

CODE — runtime/src/permissions.rs 
pub enum PermissionMode {
  ReadOnly,
  WorkspaceWrite,
  DangerFullAccess,
  Prompt,
  Allow,
}

pub struct PermissionRequest {
  pub tool_name: String,
  pub input: String,
  pub current_mode: PermissionMode,
  pub required_mode: PermissionMode,
}
白話說明

session 永遠處在這五種 authorization levels 之一。

ReadOnly：只能看，不能碰。

WorkspaceWrite：可讀寫 project folder 裡的檔案。

DangerFullAccess：完全沒有限制。

Prompt：每次敏感動作前先詢問 user。

Allow：針對某個特定動作的 override，允許它執行。

 

當 tool call 抵達時，會組出這個 bundle 來評估。

request 的是哪個 tool？（例如 "execute_bash"）

arguments 是什麼？（例如要執行的 shell command）

目前 session 在什麼 mode？

這個 tool 需要什麼 mode？如果 current < required → 擋下。

Hooks：在任何 tool 前後執行你自己的 code

plugins crate 讓你註冊 shell scripts，在 tool calls 前後自動執行。把它想成機場安檢：你設定一次，之後每個 tool call 都會經過你的 checkpoint。

⬅
PreToolUse hook

在 tool 執行前觸發。你的 script 可以檢查 tool name 和 input，然後輸出 allow、deny 或 ask 來 override permission decision。

➡
PostToolUse hook

在 tool 成功完成後觸發。你的 script 會收到 tool output，適合用來 logging、auditing，或觸發 side effects。

⚠
PostToolUseFailure hook

在 tool 失敗時觸發。讓你能對錯誤做反應：發 alert、寫 log，或嘗試 recovery。

💡
Hooks 讓你不用修改 claw 本身，也能建立 guardrails

想阻止 AI 碰任何 *.env 檔嗎？寫一個 PreToolUse hook 檢查 input，看到這個 pattern 就輸出 deny。想在 AI 每次 commit code 時發 Slack 通知嗎？在 execute_bash 上加 PostToolUse hook。這就是 power users 為團隊客製 claw 的地方。

整體圖像

Multi-provider AI、autonomous agents，以及這一切對軟體建構方式的意義

Claw 並不綁定 Claude

claw 最具策略價值的功能之一，是它能搭配 任何主要 AI provider，不只是 Anthropic。這很重要，因為 models 演進很快、價格會變，而且某個任務的最佳 model 不一定適合另一個任務。

🟠
Anthropic (Claude)

預設 provider。使用 native Anthropic Messages API，支援 streaming 和 prompt caching。設定 ANTHROPIC_API_KEY。

⚡
xAI (Grok)

Elon Musk 的 AI lab。使用 OpenAI-compatible wire format。設定 XAI_API_KEY 並使用 --model grok。

🔵
OpenAI-compatible (GPT, OpenRouter, Ollama, Qwen…)

任何支援 OpenAI Chat Completions format 的服務都能用。把 OPENAI_BASE_URL 指到任何 compatible service，包括 local models。

🦙
Local models via Ollama

完全在自己的機器上跑 model：不需要 API key、不需要網路、也沒有 per-token cost。設定 OPENAI_BASE_URL=http://localhost:11434/v1，並使用任意 Ollama model name。

🗺️
claw 如何自動選對 provider

Model name prefixes 會作為 routing signals。claude-* → Anthropic。grok-* → xAI。openai/* 或 gpt-* → OpenAI-compat。qwen-* → Alibaba DashScope。如果都不符合，就 fallback 到你 environment 裡設定的 API key。多數時候你不需要明確指定 provider。

Claw 不只是 tool，而是一場示範

包含 claw 的這個 repository，本身就是 由 AI agents 建出來的。不是輔助，而是真的建構：agents 規劃、coding、testing、reviewing、pushing，而人類從 Discord channel 提供方向。code 是證據；process 才是這個產品想教的事。

🦞

clawhip

event router。監看 git commits、tmux sessions、GitHub events 和 agent lifecycle，並在 coding agent 的 context window 外傳送 notifications。

🤝

oh-my-openagent

multi-agent coordinator。管理 planning、Architect、Executor、Reviewer agents 之間的 handoffs，以及意見不同時的 convergence。

🔁

oh-my-codex (OmX)

workflow layer。把短指令轉成結構化執行：planning keywords、parallel multi-agent workflows，以及 persistent verification loops。

?

現在真正的 bottleneck 是什麼

當 agent systems 能在幾小時內重建一個 codebase，稀缺資源就不再是打字速度。理解這件事，會改變你該如何分配時間。

🧭
方向

知道要 build 什麼，以及它為什麼重要，比知道怎麼 build 更有價值。Agents 很快；錯誤方向的代價很高。

🔬
拆解

把大目標拆成 scope 清楚、可 parallelize 的 tasks，是機器目前還不具備的能力。這就是你的槓桿。

🎯
品味

辨認 AI-generated solution「可以跑但不對」的時刻，也就是技術上能運作、架構上卻很糟，需要真正理解系統才有的 judgment。

🔤
術語

使用正確術語，例如 context window、SSE、crate、enum、permission mode、precise vocabulary，代表你能給 agents 精準、不含糊的指令。不精準的語言會導致不精準的工作。

🦞
人的工作不是比機器打字更快
