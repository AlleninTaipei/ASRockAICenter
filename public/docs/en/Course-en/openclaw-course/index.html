Meet the Router

The Router's job is to answer one question: “Which AI agent and conversation thread should handle this message?” It produces a ResolvedAgentRoute — a decision slip with everything the Gateway needs to dispatch the work.

  agentId: string;
  channel: string;
  accountId: string;
  sessionKey: string;
  mainSessionKey: string;
  lastRoutePolicy: "main" | "session";
  matchedBy:
    | "binding.peer"
    | "binding.peer.parent"
    | "binding.peer.wildcard"
    | "binding.guild+roles"
    | "binding.guild"
    | "binding.team"
    | "binding.account"
    | "binding.channel"
    | "default";
};
PLAIN ENGLISH

agentId — which AI agent handles this (you can have multiple agents)

channel — which app the message came from (telegram, whatsapp, discord…)

accountId — which bot account received it (you might have multiple bot accounts)

sessionKey — the unique ID for this conversation thread

mainSessionKey — a shortcut alias for direct (non-group) chats

lastRoutePolicy — whether to update the main session or the specific session

matchedBy — how the router found the right agent: the most useful field for debugging

"binding.peer" means an exact user-ID match — the most specific binding rule

"default" means no rules matched — the fallback agent was used

Think of this as a shipping label on every message

🔍
Debugging tip

If messages go to the wrong agent, inspect the matchedBy field in logs. A value of "default" means no binding rule matched — check your binding configuration.

Why replies appear word by word

While the AI is generating a reply, OpenClaw sends partial text to Telegram so you see it appear in real time. This is the Draft Stream Loop — a throttled loop that flushes pending text at a controlled rate.

CODE — src/channels/draft-stream-loop.ts (lines 10–50) 
export function createDraftStreamLoop(params: {
  throttleMs: number;
  isStopped: () => boolean;
  sendOrEditStreamMessage: (text: string) => Promise<void | boolean>;
}): DraftStreamLoop {
  let lastSentAt = 0;
  let pendingText = "";
  let inFlightPromise: Promise<void | boolean> | undefined;

  const flush = async () => {
    while (!params.isStopped()) {
      if (inFlightPromise) {
        await inFlightPromise;
        continue;
      }
      const text = pendingText;
      if (!text.trim()) { pendingText = ""; return; }
      pendingText = "";
      const current = params.sendOrEditStreamMessage(text).finally(() => {
        if (inFlightPromise === current) inFlightPromise = undefined;
      });
      inFlightPromise = current;
      const sent = await current;
      if (sent === false) { pendingText = text; return; }
      lastSentAt = Date.now();
      if (!pendingText) return;
    }
  };
PLAIN ENGLISH

Creates a streaming loop that sends text to the channel as the AI generates it

throttleMs = how often to send updates — prevents flooding Telegram every millisecond

isStopped() = check if the run has been cancelled before sending more

sendOrEditStreamMessage = the function that actually edits the Telegram message with new text

—

lastSentAt = timestamp of last successful send, used for throttle timing

pendingText = AI text waiting to be sent to Telegram

inFlightPromise = a Telegram edit is already in progress — wait for it before sending more

—

flush() = the core loop: grab all pending text, send it, wait, check again

If isStopped() is true — stop immediately, the run was cancelled

If a Telegram request is already flying — wait for it to finish, then loop again

Grab all pending text, then clear the buffer so new AI text can accumulate

Send the text to Telegram and track the request as “in flight”

If sent === false: Telegram rejected this batch — hold onto the text and retry next cycle

Update lastSentAt, then check if more text arrived while we were waiting

⚡
Why throttle?

Telegram has a rate limit on how many times you can edit a message per second. The throttleMs parameter ensures OpenClaw never hits that limit, even during a fast-streaming response.

The “Do Not Disturb” system

While a reply is being generated, OpenClaw tracks whether each conversation is busy. This prevents a second message from interrupting a reply that is still being written.

CODE — src/channels/run-state-machine.ts (lines 18–34) 
export function createRunStateMachine(params: RunStateMachineParams) {
  const heartbeatMs = params.heartbeatMs ?? DEFAULT_RUN_ACTIVITY_HEARTBEAT_MS;
  let activeRuns = 0;
  let runActivityHeartbeat: ReturnType<typeof setInterval> | null = null;

  const publish = () => {
    params.setStatus?.( {
      activeRuns,
      busy: activeRuns > 0,
      lastRunActivityAt: now(),
    });
  };
PLAIN ENGLISH

Creates a state tracker for one conversation thread

heartbeatMs = send a status update this often while a run is active

activeRuns = how many AI runs are currently happening (normally 0 or 1)

runActivityHeartbeat = a repeating timer that fires status updates while busy

—

publish() = broadcasts the current state to anyone listening

activeRuns = the live count of ongoing AI calls in this thread

busy: activeRuns > 0 = true means a reply is in progress — the channel queues the next message

lastRunActivityAt = timestamp of the last activity, used for health monitoring

Like an “on air” light in a radio studio — red means someone is talking, wait your turn

🔴
Busy (activeRuns > 0)

A reply is being generated. New messages are queued and delivered after the current run completes.

🟢
Idle (activeRuns === 0)

No reply in progress. The next message triggers a new inference loop immediately.

💓
Heartbeat

While busy, a periodic pulse is sent so the Gateway knows the run is still alive — not silently crashed.

The Brain Inside

The Pi Agent holds your conversation history, decides which AI model to call, runs tools like a browser or bash terminal, and streams the answer back. Here is how it thinks.

Your conversation, on disk

A session is a list of messages — yours and the AI’s — stored as a JSONL file on your disk. Every new message is appended to that file, and before each AI call, the entire history is bundled and sent to the model as context.

📜
Session History

The full conversation as a JSONL file. The AI gets to read every prior message — yours and its own — before generating each reply.

🔑
Session Key

The unique ID for this conversation (from the Router). The Pi Agent uses it to find the right file on disk.

✂️
Compaction

When the history grows too large for the AI to handle, the oldest messages are summarized into a compact synopsis. Like condensing a transcript into meeting notes.

🗂️
Parallel Sessions

Each channel + peer combination gets its own session. You can have hundreds of separate AI conversations running simultaneously — they never interfere.

If one AI fails, try the next

OpenClaw supports Claude, GPT, Gemini, Llama, and dozens of other models. If a provider is rate-limited or down, it automatically tries the next configured model — a mechanism called model failover.

CODE — src/agents/model-fallback.ts (lines 45–60) 
export class FallbackSummaryError extends Error {
  readonly attempts: FallbackAttempt[];
  readonly soonestCooldownExpiry: number | null;

  constructor(
    message: string,
    attempts: FallbackAttempt[],
    soonestCooldownExpiry: number | null,
    cause?: Error,
  ) {
    super(message, { cause });
    this.name = "FallbackSummaryError";
    this.attempts = attempts;
    this.soonestCooldownExpiry = soonestCooldownExpiry;
  }
}
PLAIN ENGLISH

FallbackSummaryError = the error thrown only when every configured model has been tried and failed

attempts = a log of each model tried and why it failed (rate limit? timeout? auth error?)

soonestCooldownExpiry = the earliest time any model will accept requests again

—

message = a human-readable summary of all the failures

attempts = the full list of what was tried, in order

soonestCooldownExpiry = when to retry (milliseconds timestamp)

cause = the last underlying error, for debugging purposes

—

Like a rejection slip with full notes: tried Claude (rate-limited), tried GPT (timed out), tried Gemini (auth error). Try again in 47 seconds.

1️⃣
Try primary model

The first configured provider (usually Claude or GPT) gets the request.

❌
Primary fails

Rate-limited, timed out, or auth error. Record the attempt and move to the next provider.

2️⃣
Try next provider

Each fallback is tried in sequence until one succeeds or all are exhausted.

🚫
All failed

Throw FallbackSummaryError with the full attempt log and the soonest retry time.

The whiteboard that fills up

Every AI model has a limit on how much text it can read at once — called the context window. Think of it like a whiteboard: you can only fit so much writing before you have to erase the oldest notes.

💡

“Context window” is one of the most important concepts in AI. It explains why your AI sometimes “forgets” something from earlier in a long conversation — the old messages fell off the whiteboard. When you tell an AI agent to “improve the bot’s memory”, you are asking it to work with the context window more cleverly.

📏
Token budget

Each model has a fixed context window — typically 128k–200k tokens for modern models. Longer conversations eat into this budget faster.

🗜️
Compaction triggers

When the history approaches the limit, compaction summarizes the oldest messages into a brief synopsis and replaces them.

📝
What gets kept

The synopsis captures the gist of early conversation. Specific details (names, numbers, exact instructions) may be lost — this is a known tradeoff.

How plugins introduce themselves

Every plugin declares itself through a manifest file — a JSON file that announces who you are and what you provide. The Gateway reads all manifests at startup and builds its plugin registry without loading any code yet.

openclaw.plugin.json — the plugin identity card
"id" Unique plugin identifier — e.g. "matrix", "zalo-user", "anthropic-vertex"
"openclaw.channel.id" If it is a channel plugin, the channel identifier used in routing config
"openclaw.install.npmSpec" The npm package name used for installation — e.g. "@openclaw/matrix"
"entrypoint" The JavaScript file to execute when the plugin loads — only read after the manifest is approved
💡
Manifest-first architecture: discover with metadata, execute only when needed

Reading metadata is fast. Loading and running code is slow. By reading manifests first, the Gateway can show you all installed plugins in the Control UI instantly — before any plugin code actually runs.

  const raw = config?.skills?.install;

Staying Healthy

OpenClaw runs 24/7 on your device. Here is how it stays healthy: it watches for broken connections, reloads config without restarting, locks down untrusted senders, and diagnoses its own problems.

Four systems keeping OpenClaw alive

Think of OpenClaw as a well-designed ship. The captain monitors all systems, the health officer watches each port connection, the locksmith controls who boards, and the engineer runs diagnostics and fixes mechanical issues. Four distinct subsystems work together to keep everything running smoothly.

🫀
Channel Health Monitor

Continuously probes each connected channel. If a channel goes quiet for too long, it triggers a reconnect — automatically, without any action from you.

🔄
Config Hot Reload

You can edit your config file and OpenClaw applies the changes instantly — no restart needed. Like editing a live playlist without stopping the music.

🔒
Security Defaults

Every DM from an unknown sender triggers a pairing challenge. They get a code; you approve it. No random strangers get an AI assistant by default.

🩺
The Doctor

openclaw doctor inspects your entire setup: config, credentials, channel connections, legacy state. Add --fix and it auto-repairs many problems — your first stop after any upgrade.

🚢
A well-designed ship analogy

The health monitor watches every port (channel) and reconnects if a connection drops. The security policy controls who boards (DM pairing). The hot-reload engine applies config changes without docking (restarting). The doctor runs full diagnostics on demand.

The always-on heartbeat checker

Every channel has a health policy that defines how aggressively to monitor its connection. If a channel goes silent for too long, the monitor triggers a reconnect.
