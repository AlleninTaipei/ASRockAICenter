What OpenCode Actually Is

OpenCode is an open-source AI coding agent. You describe what you want in plain language, and it reads files, writes code, and runs tests on your behalf.

Built by the creators of terminal.shop, OpenCode works with Claude, GPT-4, Gemini, and local models.

03

Anatomy of a Tool

Every tool OpenCode can call is defined by a TypeScript interface in the source code. Here is what that contract looks like — and what it means in plain English.

CODE 
export interface Def<Parameters extends z.ZodType = z.ZodType,
     M extends Metadata = Metadata> {
  id: string
  description: string
  parameters: Parameters
  execute(args: z.infer<Parameters>,
          ctx: Context): Effect.Effect<ExecuteResult<M>>
  formatValidationError?(error: z.ZodError): string
}
PLAIN ENGLISH

id: string — A unique name for this tool, like "bash", "read", or "webfetch". The LLM uses this ID to call the right tool.

description: string — Plain English explanation so the AI knows when to use this tool. This is literally what the AI reads to decide.

parameters: Parameters — The inputs this tool accepts — like "command" and "timeout" for bash. Validated with Zod before execution.

execute(args, ctx) — The actual function that runs when the AI calls this tool. Returns an Effect.Effect — OpenCode's way of handling async operations safely.

formatValidationError? — Optional: a custom error message if the AI passes wrong inputs. The ? means this field is not required.

💡
Why This Design Matters

Every tool is a contract between the AI and your computer. The AI reads the description to know WHEN to use it. The parameters define HOW to use it. This separation is what makes AI agents safe and predictable — the AI cannot call a tool incorrectly if the schema rejects invalid inputs.

📄
Where to find this in the source

Source file: packages/opencode/src/tool/tool.ts. Every built-in tool — bash, read, edit, grep, webfetch — implements this same interface.

Module 2 of 5

The Brain Behind the Magic

How AI agents make decisions — and why there are three of them

OpenCode does not have one AI — it has a team of them with different personalities and different levels of trust. Here is why that matters.

The multi-agent system exists because different tasks require different risk tolerances. A planning AI that cannot accidentally delete files is safer for exploration. A building AI that can run any command is more powerful — but needs more trust.

Not One AI — A Team

Every time you use OpenCode, you are working with one of three specialised agents, each with different capabilities and constraints.

🏗️

Build Agent

Full permissions — can read files, write code, run bash commands. This is your hands-on contractor. Used for most tasks.

📋

Plan Agent

Read-only mode — can explore and analyze but cannot change anything. Your safe research assistant. Great for understanding a codebase.

🔍

General Agent

Spawned automatically as a subagent for complex searches. Runs in the background to gather information for the main agent.

🎯
Practical Rule of Thumb

The agent you are using affects what OpenCode is allowed to do. When you are not sure what to ask for — start with Plan mode. When you are ready to make changes — switch to Build.

The Reasoning Loop

The LLM does not just answer once. It thinks in a loop — calling tools, reading results, and deciding whether it needs more information before giving a final answer.

1
Receive context

Your message + conversation history + the list of available tools are all loaded into the AI at once.

2
Decide

What tool (if any) would help me answer this? The AI weighs its options and picks the most useful next action.

3
Act

Call the tool, then wait for the result. This is a real tool call: the AI pauses and the runtime executes it.

4
Observe

What did the tool return? Does this change my plan? The AI reads the output and updates its understanding.

5
Loop or Finish

Go back to step 2 if more information is needed — or write the final answer if the task is complete.

💡
Engineers call this a "ReAct Loop"

ReAct loop stands for Reasoning + Acting. It is the core pattern behind every AI agent, not just OpenCode. When you see the AI calling multiple tools before answering, this loop is exactly what you are watching.

The Context Window Problem

Every model has a memory limit called the context window. It can only hold so many words at once.

🗒️
The Whiteboard Metaphor

Imagine a whiteboard that holds 200 sticky notes. Fill it up and you must erase the oldest ones to add new ones — no matter how important they were. That is exactly how the context window works.

Here is the real code that describes everything sent to the AI for each response. It comes from packages/opencode/src/session/llm.ts:

CODE — StreamInput type 
export type StreamInput = {
  user: MessageV2.User
  sessionID: string
  parentSessionID?: string
  model: Provider.Model
  agent: Agent.Info
  permission?: Permission.Ruleset
  system: string[]
  messages: ModelMessage[]
  small?: boolean
  tools: Record<string, Tool>
  retries?: number
  toolChoice?: "auto" | "required" | "none"
}
PLAIN ENGLISH

user — The message you just sent — your most recent request

sessionID — Which conversation this belongs to

model — Which AI brain to use (Claude, GPT-4, Gemini, etc.)

agent — Which agent mode is active (build, plan, or general)

permission — The safety rules for what this agent is allowed to do

system — The instructions that define how the AI should behave (the system prompt)

messages — THE KEY FIELD: the conversation history sent to the AI — this is what fills up the context window

tools — The full list of tools the AI can call this turn

toolChoice — Whether the AI must use a tool, can choose, or is prevented from using tools at all

🗜️
Compaction: OpenCode's Memory Management

When a session gets very long, OpenCode runs compaction — it summarizes the older parts of the conversation into a shorter summary, then continues. You lose some detail but gain space for more messages.

Each agent is defined by its own configuration object. Here is what that looks like in packages/opencode/src/agent/agent.ts:

CODE — Agent.Info schema 
export const Info = z
  .object({
    name: z.string(),
    description: z.string().optional(),
    mode: z.enum(["subagent", "primary", "all"]),
    native: z.boolean().optional(),
    permission: Permission.Ruleset.zod,
    model: z.object({
      modelID: ModelID.zod,
      providerID: ProviderID.zod,
    }).optional(),
    prompt: z.string().optional(),
    steps: z.number().int().positive().optional(),
  })
PLAIN ENGLISH

name — The agent name — "build", "plan", or a custom name you define

mode — How this agent is used: as the main agent, a background helper (subagent), or both

permission — The permission ruleset this agent must follow — the core safety constraint

model — Optional: override which AI model this specific agent uses, independent of your global setting

prompt — Optional custom system prompt to change this agent's personality or instructions

steps — Maximum number of tool calls before stopping — a safety limit to prevent runaway loops

🔑
Why Zod?

The z.object() syntax comes from Zod schema — a validation library for TypeScript. It means OpenCode validates agent configurations at runtime — if you define a custom agent with bad values, it fails immediately with a clear error rather than silently misbehaving.

Anatomy of a Tool — The Bash Example

Every tool is defined by a schema — a precise description of what inputs it accepts. Here is the real schema for the bash tool from OpenCode source code, alongside a plain English translation.

CODE 

const Parameters = z.object({
  command: z.string()
    .describe("The command to execute"),
  timeout: z.number()
    .describe("Optional timeout in milliseconds")
    .optional(),
  workdir: z.string()
    .describe("The working directory to run
     the command in...")
    .optional(),
  description: z.string()
    .describe("Clear, concise description
     of what this command does..."),
})
PLAIN ENGLISH

command — The actual command to run — like npm test or git status

timeout (optional) — A time limit in milliseconds — stop the command if it takes too long

workdir (optional) — Which folder to run the command in — like running npm install inside a specific package folder

description — A plain English description of what this command does — helps you review what the AI is about to run

💡
Why description is required — not optional

This forces the AI to explain its reasoning before every bash command. You can always see WHY it wants to run something, not just WHAT it is running.

03

The Permission System — Your Safety Net

The permission system gives you fine-grained control over every tool call. Three actions are available:

✅
allow

Run without asking — you have pre-approved this action

⏸️
ask

Pause and show a confirmation prompt — you decide in the moment

🚫
deny

Block completely — this action is never allowed

Example rules in JSON configuration

bash → * → ask Must confirm all bash commands before they run — the wildcard catches everything
bash → npm test → allow Running tests is pre-approved — no confirmation needed for this specific command
edit → *.env* → ask Ask before editing any .env file — the glob pattern *.env* matches any env file
edit → src/** → allow Free to edit anything in the src/ folder — the AI works uninterrupted on source code
bash → rm -rf → deny Block this dangerous command entirely — it can never run, no matter what
📋
Most specific rule wins

Rules are checked in order — the most specific match takes precedence. A rule for npm test is more specific than a rule for *, so it overrides the wildcard.

Extending with MCP Servers

What is MCP? The Model Context Protocol is an industry standard for connecting external tools to AI. Like a USB port — any tool built to the spec plugs into any AI that supports it.

MCP servers can wrap internal tools, APIs, databases, and services — exposing them to the AI through the same tool interface as built-in capabilities.

🔌
Automatic Discovery

The AI discovers MCP tools automatically at startup — they appear alongside the 43 built-in tools. From the AI perspective, there is no difference between a built-in tool and an MCP tool.

Example MCP servers

🐙
GitHub MCP

Adds tools for reading PRs, creating issues, searching commits — the AI can navigate your GitHub repo directly

💬
Slack MCP

Adds tools for posting messages and reading channels — the AI can notify your team when it finishes a task

🏢
Custom MCP

Build your own server to expose any internal tool — deploy scripts, database queries, monitoring dashboards

How to configure an MCP server

command: mcp-github The command to start the MCP server process — OpenCode launches this when it starts up
env: GITHUB_TOKEN Authentication credentials passed to the server — kept separate from your code

02

The Session Table — Code Decoded

Every session you open becomes one row in the session table of the schema. Here is the real code from OpenCode, translated into plain English.

CODE 

export const SessionTable = sqliteTable(
  "session",
  {
    id: text().primaryKey(),
    project_id: text()
      .references(() => ProjectTable.id),
    workspace_id: text(),
    parent_id: text(),
    slug: text().notNull(),
    directory: text().notNull(),
    title: text().notNull(),
    share_url: text(),
    summary_additions: integer(),
    summary_deletions: integer(),
    summary_files: integer(),
    summary_diffs: text({ mode: "json" }),
    revert: text({ mode: "json" }),
    permission: text({ mode: "json" }),
  }
)
PLAIN ENGLISH

id — A unique ID for this session — like a receipt number. This is the primary key.

project_id — Which project folder this session belongs to.

workspace_id — Optional: which enterprise workspace this belongs to.

parent_id — If you forked from another session, this points to the original.

slug — A short URL-friendly name for the session.

directory — The folder on your computer where OpenCode was running.

title — The display name for this session (shown in the UI).

share_url — If you shared this session, the encrypted link is stored here.

summary_additions / deletions / files — How many lines were added, deleted, and how many files changed.

summary_diffs — A JSON summary of every file that was changed.

revert — The checkpoint data needed to undo everything back to this point.

permission — The permission rules that were active during this session.

💡
Relational Databases in 30 Seconds

Storing data in structured tables like this is called a relational database. Think of it like a spreadsheet where each row is one session and each column is a piece of information. SQLite is a tiny, fast database that lives as a single file on your computer — no server needed.

OpenCode uses Drizzle ORM to define these tables in TypeScript, making the migrations type-safe and keeping the blob fields strongly typed.

03

The Configuration Hierarchy

OpenCode reads settings from three layers, each overriding the one below it. Understanding this hierarchy tells you exactly where to put any setting.

1
System / Enterprise (MDM)

Managed by IT via Apple MDM or Windows GPO. Cannot be overridden by users. Used in enterprise deployments to enforce company-wide settings like approved AI providers.

2
User-level (~/.opencode/config.jsonc)

Your personal defaults across all projects. This is where you set your preferred AI model, default permissions, and personal MCP servers.

3
Project-level (.opencode/config.jsonc)

Project-specific settings that override your personal defaults. This is where you set which model is best for this codebase, project-specific permissions, and team-shared MCP servers.

Where the files live

~/ your home folder
.opencode/ personal OpenCode folder
config.jsonc your personal settings — applies to all projects
your-project/ any project folder
.opencode/ project-level OpenCode folder
config.jsonc project settings — overrides personal settings

Key config options

model Which AI to use: claude-sonnet-4-5, gpt-4o, gemini-1.5-pro, etc.
provider Which AI provider: anthropic, openai, google, local, etc.
permission Your allow / deny / ask rules for tools and file paths.
mcp External tool servers to connect at startup.
plugins npm packages that add new tools or UI features.
📄
Why .jsonc instead of .json?

JSONC is JSON with Comments — the c stands for comments. It lets you add notes like // my work laptop API key right inside the config file, which regular JSON does not allow.

How Real-Time Updates Work

WebSocket is like a phone call — once you dial, the line stays open and either side can speak at any moment. HTTP requests are like letters — you send one, wait for a reply, then the connection closes.

When the AI is working, every event — a file edit, a tool call, a message — is broadcast over WebSocket to every connected client simultaneously.

The AI fires one event. The event bus carries it. Every client listening on the WebSocket reacts instantly — terminal, browser, desktop app, all at once.

03

The Event Bus — Code Decoded

The event bus is defined in packages/opencode/src/bus/bus.ts. Here is the real interface — and what each method does in plain English.

CODE — Bus.Interface 
export interface Interface {
  readonly publish: <D extends BusEvent.Definition>(
    def: D,
    properties: z.output<D["properties"]>,
  ) => Effect.Effect<void>
  readonly subscribe: <D extends BusEvent.Definition>(def: D) => Stream.Stream<Payload<D>>
  readonly subscribeAll: () => Stream.Stream<Payload>
  readonly subscribeCallback: <D extends BusEvent.Definition>(
    def: D,
    callback: (event: Payload<D>) => unknown,
  ) => Effect.Effect<() => void>
  readonly subscribeAllCallback: (callback: (event: any) => unknown) => Effect.Effect<() => void>
}
PLAIN ENGLISH

publish(def, properties) — Send an event to all listeners — like a radio tower broadcasting a signal

subscribe(def) — Listen for one specific type of event — like tuning to a specific radio frequency

subscribeAll() — Listen for ALL events — useful for debugging or logging everything that happens

subscribeCallback(def, callback) — Listen for a specific event and run a function when it arrives

subscribeAllCallback(callback) — Run a function every time ANY event fires — the most powerful listener

🔗
The Secret to Staying in Sync

This interface is what lets the terminal UI, web UI, and VS Code extension all stay in sync without talking to each other directly. They all listen to the same bus. The bus carries the truth.

04

Building on Top of OpenCode

Because OpenCode exposes a standard REST API and WebSocket, you can build anything on top of it. Here are three real-world use cases.

💬
Use Case 1: Slack Bot

Send a message to Slack with a coding task → your bot calls POST /sessions to start a session → POST /sessions/{id}/messages to send the task → WebSocket stream returns the result → bot replies in Slack with the outcome.

🔄
Use Case 2: CI/CD Integration

After every pull request → your pipeline starts an OpenCode session → asks it to review the diff for security issues → gets a structured response → posts it as a PR comment.

🎨
Use Case 3: Custom Dashboard

Build any frontend you want — React, Vue, plain HTML — that connects to the REST API. Your own branding, your own UI, same powerful engine underneath. It is headless by design.

The 4 key API endpoints

POST /sessions Create a new session in a project directory — this is where every interaction begins
POST /sessions/{id}/messages Send a message and start the AI working — the core interaction endpoint
GET /sessions/{id}/messages Retrieve the full conversation history — useful for displaying past sessions or building summaries
WS /sessions/{id}/subscribe Open a WebSocket to receive live events as they happen — the real-time stream of everything the AI is doing
