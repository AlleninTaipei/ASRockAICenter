Autoresearch: How AI Teaches Itself
01

The Overnight Lab

You press one button before bed. By morning, an AI has run 100 experiments — and gotten smarter each time.

What just happened while you slept?

Imagine leaving a very determined intern in a lab overnight. Their job: try every idea they can think of to improve a machine. They keep what works, throw out what doesn't, and never stop — even at 3am.

Autoresearch is that intern, but the machine is an AI — and the AI is experimenting on itself.

1
The AI agent reads the training code

It looks at train.py — the file that defines how the model learns — and decides what to change.

2
It runs a 5-minute experiment

The modified code trains a small language model on real text data for exactly 300 seconds.

3
It checks the score

One number — val_bpb — tells if the model improved. Lower is better.

4
Keep or discard — then repeat forever

If the score improved, keep the change and continue. If not, revert and try something else.

Why is this a big deal?

Normally, AI research takes teams of PhD scientists months — reading papers, designing experiments, waiting for results. Autoresearch compresses that cycle to 5 minutes per experiment.

⚡

~12 experiments/hour

While you sleep 8 hours, the agent runs roughly 100 experiments — more than most human researchers run in a month.

🎯

One metric to rule them all

Every experiment is judged by the same single score. No subjective judgment — just "did it improve?"

🔁

Self-directed research

The AI decides what to try next. It reads its own code, forms a hypothesis, and tests it — without human guidance on each step.

💡

You are not building a better model directly. You are building an environment where better models emerge automatically. The agent does the research; you design the rules of the game.

The Three Files

The entire project lives in three files — and each one has a completely different job.

Three files, three roles

Think of this project like a theatrical production. There is the stage crew (invisible but essential), the performer (what the audience sees change), and the director's script (the instructions that guide everything). Each file plays exactly one role — and they never swap.

🔧
prepare.py — The Stage Crew

Downloads data, trains the tokenizer, builds the dataloader, and runs the final evaluation. You never modify this file. It is locked.

✏️
train.py — The Performer

The only file the agent edits. Contains the full GPT model architecture, the optimizer, and the training loop. Every experiment is a new version of this file.

📋
program.md — The Director's Script

A Markdown file written in plain English that tells the AI agent exactly what to do, how to evaluate results, and what rules to follow.

The project structure

Here is the complete project. Three core files and almost nothing else — intentionally minimal so the agent can focus.

autoresearch/ The whole project
prepare.py 🔒 Read-only — data pipeline & evaluation harness
train.py ✏️ Agent modifies this — model, optimizer, training loop
program.md 📋 Human modifies this — instructions for the AI agent
pyproject.toml Dependencies list — also locked, no new packages allowed
results.tsv Experiment log — tab-separated, written by agent after each run
🔒
Why lock prepare.py?

If the agent could change the evaluation, it could cheat — like a student changing the answer key instead of studying. Locking prepare.py guarantees every experiment is judged by the same fair standard.

The fixed rules — prepare.py constants

Three constants in prepare.py set the ground rules for every experiment. No experiment can change them.

CODE 

# Constants (fixed, do not modify)
MAX_SEQ_LEN = 2048      # context length
TIME_BUDGET = 300       # training time in seconds
EVAL_TOKENS = 40 * 524288 # ~20M tokens for val eval
PLAIN ENGLISH

These three numbers are the rulebook — every experiment plays by them.

MAX_SEQ_LEN = 2048: The model can read 2,048 tokens at once — roughly 1,500 words of context.

TIME_BUDGET = 300: Training always stops after exactly 5 minutes. No more, no less.

EVAL_TOKENS: The test uses about 20 million tokens of text the model has never seen. More tokens = more reliable score.

03

Teaching Computers to Read

Inside train.py lives a GPT model — a system that learns to predict text by seeing billions of examples.

First: turning words into numbers

Computers only understand numbers. So before any learning can happen, every word must be converted into a number. This is the job of the tokenizer.

The tokenizer in this project breaks text into small chunks — usually whole words or common word-pieces — and assigns each one a number from 0 to 8,191 (the vocabulary size is 8,192).

🔤
Example: "Hello world" → [4521, 1204]

The word "Hello" maps to token 4521. "world" maps to token 1204. The model only ever sees these numbers — never raw text. The tokenizer is the translation layer between human language and machine numbers.

Once text is converted to token numbers, it flows into the model layer by layer — like an assembly line where each station adds a bit more understanding.

The GPT assembly line

The model transforms raw token numbers into a prediction of "what comes next" through a series of layers. Each layer refines the understanding a little more.

The forward pass in code

The forward pass is the core computation. Here is the key section of the GPT.forward() method from train.py:

CODE 

def forward(self, idx, targets=None):
    x = self.transformer.wte(idx)
    x = norm(x)
    x0 = x
    for i, block in enumerate(self.transformer.h):
        x = self.resid_lambdas[i] * x + self.x0_lambdas[i] * x0
        x = block(x, ve, cos_sin, window_sizes[i])
    logits = self.lm_head(x)
PLAIN ENGLISH

This function takes token numbers in, returns predictions out.

Look up each token in the embedding table — convert numbers to rich vectors of meaning.

Normalize the vectors so they have consistent scale (prevents runaway math).

Save a copy of the starting representation — we will use it as an anchor later.

Run the data through each of the 8 transformer blocks in sequence.

Blend the current representation with the original anchor — helps the model remember where it started.

Each block (attention + MLP) refines the understanding one level deeper.

Convert the final representation into prediction scores for every possible next token.

The clever tricks inside the model

This is not a plain GPT. It has several engineering improvements baked in. Think of them as upgrades to a standard camera — same basic design, better optics.

🌀

Rotary Embeddings

Instead of adding a position number to each token, the position is "rotated" into the token's meaning vector. This lets the model handle sequences it was never trained on.

🪟

Sliding Window Attention

Most layers only look at nearby tokens (half the context window). This is faster and works surprisingly well — like reading a paragraph without looking back to page 1 every time.

📐

RMSNorm

Normalization keeps the numbers at a sane scale throughout training. RMSNorm is a simpler, faster version of LayerNorm — it scales without centering.

🎚️

Value Embeddings (ResFormer)

Every other layer gets a direct shortcut from the original token embeddings into the attention values. This helps the model remember what tokens meant at the start.

💡
Key Insight: You do not need to understand these to steer the AI

The agent can try adding or removing these features by changing a few lines in train.py. Your job as the human is to set the rules in program.md — the agent figures out which tricks help.

04

The 5-Minute Race

How does a model actually learn? And why does the training stop at exactly 300 seconds?

Learning is just adjusting numbers

The model contains millions of numbers — its weights. At the start, these numbers are random. Training adjusts them one tiny step at a time until the model gets good at predicting text.

The core loop is simple: show the model some text, ask it to predict the next word, measure how wrong it was, then adjust the weights to be a little less wrong next time.

1
Feed: show the model a batch of text

128 sequences of 2,048 tokens each — about 262,000 tokens per step.

2
Predict: model guesses the next token at every position

For each of the 2,048 positions, the model outputs 8,192 probability scores.

3
Measure: calculate the loss

Loss is the average surprise — how far off were the predictions from the real answers?

4
Adjust: backpropagation updates the weights

Math figures out which weights caused the errors and nudges them in the right direction. This is gradient descent.

5
Repeat — hundreds of times in 5 minutes

The training loop runs about 900 steps in 300 seconds on an H100 GPU.

The training loop in code

Here is the heart of train.py — the main training loop. It keeps running until the 5-minute clock runs out:

CODE 

while True:
    for micro_step in range(grad_accum_steps):
        with autocast_ctx:
            loss = model(x, y)
        loss.backward()
        x, y, epoch = next(train_loader)
    optimizer.step()
    if total_training_time >= TIME_BUDGET:
        break
PLAIN ENGLISH

Keep looping until we decide to stop.

For each mini-batch (we accumulate several before updating)...

Use efficient bfloat16 math to save memory and speed things up.

Run the model on input x, compare to target y, get the loss.

Compute gradients — figure out how each weight contributed to the error.

Load the next batch of text from the dataloader while we compute.

Update all model weights using the accumulated gradients.

Check if 300 seconds of training time have passed.

If time is up, exit the loop.

Two optimizers for different jobs

Most models use one optimizer. This project uses two — one for big weight matrices, one for everything else. Think of it like a construction crew: different tools for different materials.

🏗️
Muon — for the big matrices (attention + MLP weights)

Muon uses a mathematical trick called orthogonalization to make gradient updates more efficient. It is especially good for the large 2D weight matrices in transformer layers.

🔧
AdamW — for embeddings, biases, and scalars

AdamW is the workhorse of modern AI training — it adapts the learning rate per parameter. Used here for embeddings and scalars where Muon would be overkill.

What does val_bpb actually measure?

Every experiment ends with one number: val_bpb. Understanding this number helps you make smarter decisions about what to change.

📐
Bits Per Byte (BPB) — a vocabulary-independent measure

BPB measures how many bits of information the model needs to encode each byte of text. A perfect model would need 0 bits (it predicts everything correctly). A random model needs about 8 bits per byte. Lower is better.

The key word is validation — val_bpb is measured on text the model has never seen during training. This prevents overfitting — a model that simply memorized training text would score terribly on the validation set.

CODE 

def evaluate_bpb(model, tokenizer, batch_size):
    steps = EVAL_TOKENS // (batch_size * MAX_SEQ_LEN)
    total_nats = 0.0
    total_bytes = 0
    for _ in range(steps):
        loss_flat = model(x, y, reduction='none').view(-1)
        nbytes = token_bytes[y_flat]
        total_nats += (loss_flat * mask).sum().item()

This function calculates the final score on unseen data.

How many evaluation steps needed to cover ~20 million tokens?

Track total loss in nats (a unit for information).

Loop through all the validation batches.

Divide total surprise by total bytes, convert to bits. That is val_bpb.

Warming up and cooling down

The learning rate — how big each weight adjustment is — is not constant. It follows a schedule, like a runner who warms up before a race and slows down near the finish.

WARMUP_RATIO = 0.0 No warmup phase (currently disabled — training starts at full learning rate immediately)
WARMDOWN_RATIO = 0.5 In the last 50% of training time, the learning rate gradually decreases to zero — letting the model settle on its best weights
FINAL_LR_FRAC = 0.0 By the end of training, the learning rate reaches zero — the model stops changing at the very end
💡
Key Insight: Learning rate is one of the most impactful knobs

Too high: the model overshoots and never converges. Too low: training is slow and inefficient. The warmdown trick — decaying the LR in the second half — is one of the most reliable ways to squeeze extra performance out of a fixed time budget.

Steering the AI

Now that you understand the system, here is how to actually influence what it does — and what to watch for when things go wrong.

Your control panel: the hyperparameters

All the dials you can turn in train.py are listed at the top of the file as plain variables. Think of it like a cockpit: each lever controls a specific system, and they interact with each other in non-obvious ways.

DEPTH = 8 Number of transformer layers. Higher = smarter but slower. The primary knob for model capacity.
ASPECT_RATIO = 64 Controls model width relative to depth. model_dim = depth × aspect_ratio. Bigger = more parameters per layer.
TOTAL_BATCH_SIZE = 2**19 Total tokens processed per optimizer step (~524K). Larger batches are more stable but take longer per step.
MATRIX_LR = 0.04 Learning rate for the Muon optimizer (weight matrices). The most impactful LR to tune.
WARMDOWN_RATIO = 0.5 Fraction of training time to spend decaying the learning rate. Higher = more time settling.
WINDOW_PATTERN = "SSSL" Attention window pattern per layer. S = short (half context), L = full context. SSSL = 3 short + 1 full, repeating.

When things go wrong — fast-fail protection

The training loop has two built-in safety checks that abort immediately if something goes badly wrong. Understanding them helps you debug when the agent hits a wall.

CODE 

# Fast fail: abort if loss is exploding or NaN
if math.isnan(train_loss_f) or train_loss_f > 100:
    print("FAIL")
    exit(1)
PLAIN ENGLISH

After every training step, check two failure conditions.

NaN (Not a Number): the math produced a nonsense result — usually from a learning rate that is too high or a numerical instability.

Loss above 100: the model is catastrophically wrong — something fundamental broke. Normal loss starts around 3-5 and improves.

If either happens: print FAIL and crash immediately. Do not waste 5 minutes on a broken run.

⚠️
Debugging tip: NaN usually means the learning rate is too high

If you see FAIL or NaN in the log, the first thing to try is halving MATRIX_LR or EMBEDDING_LR. The model is taking steps so large it is flying off the rails.
