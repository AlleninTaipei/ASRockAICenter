What Is the Agent Harness? UNIVERSAL

The term was formalized in early 2026, but the concept existed long before. The harness is the complete software infrastructure wrapping an LLM: orchestration loop, tools, memory, context management, state persistence, error handling, and guardrails.

Agent The emergent behavior — the goal-directed, tool-using, self-correcting entity the user interacts with. When someone says "I built an agent," they mean they built a harness and pointed it at a model.
Harness The machinery producing agent behavior. Anthropic's Claude Code docs: "the SDK is the agent harness that powers Claude Code." OpenAI's Codex explicitly equates "agent" and "harness" to refer to the non-model infrastructure.
Context Window Serves as RAM — fast but limited. Key information must be actively managed within this scarce resource.
External Databases Function as disk storage — large but slow. Provides persistence across sessions.
Tool Integrations Act as device drivers — connect the model to the outside world via defined schemas.
The Harness itself Is the operating system. Beren Millidge (2023): "We have reinvented the Von Neumann architecture" — a natural abstraction for any computing system built around a stateless compute core.
EVIDENCE

LangChain changed only the infrastructure wrapping their LLM — same model, same weights — and jumped from outside the top 30 to rank 5 on TerminalBench 2.0. A separate research project hit 76.4% pass rate by having an LLM optimize the infrastructure itself, surpassing hand-designed systems.

"If you're not the model, you're the harness." — LangChain's Vivek Trivedy. A raw LLM is a CPU with no RAM, no disk, and no I/O. The harness is the operating system that makes it useful.

Three Levels of Engineering UNIVERSAL

Three concentric levels surround the model. Each level builds on the previous; the harness encompasses all of them.

1 · Prompt Engineering
2 · Context Engineering
3 · Harness Engineering

Prompt Engineering

Crafts the instructions the model receives at each turn. The innermost layer — focused on what to say to the model. Effective prompts position critical instructions at the beginning and end of the context window to counter the "Lost in the Middle" effect.

SCOPE

Operates at a single-turn level. Does not address what happens across turns, what data to include, or what the model should do after generating output.

Context Engineering

Manages what the model sees and when — controlling the entire information landscape across turns. Goes beyond individual prompts to handle retrieval, compression, and prioritization of context.

SCOPE

Anthropic's context engineering goal: find the smallest possible set of high-signal tokens that maximize likelihood of the desired outcome. Context rot — performance degrades 30%+ when key content falls in mid-window positions (Stanford's "Lost in the Middle").

Harness Engineering

Encompasses both prompt and context engineering, plus the entire application infrastructure: tool orchestration, state persistence, error recovery, verification loops, safety enforcement, and lifecycle management.

SCOPE

The harness is not a wrapper around a prompt. It is the complete system that makes autonomous agent behavior possible — deciding when to call the model, what tools to expose, how to recover from failures, and when to stop.

The Orchestration Loop — The Heartbeat

Implements the Thought-Action-Observation (TAO) cycle, also called the ReAct loop. Mechanically it's often just a while loop. The complexity lives in everything the loop manages, not the loop itself.

while not_done:
    prompt  = assemble_prompt(context, memory, tools)
    output  = call_llm(prompt)
    actions = parse_tool_calls(output)
    if actions:
        results = execute_tools(actions)
        append_to_context(results)
    else:
        return output  # final answer — no more tool calls
ANTHROPIC

Anthropic describes their runtime as a "dumb loop" where all intelligence lives in the model. The harness just manages turns — it does not reason, plan, or decide. Those are the model's jobs.

Tools — The Agent's Hands

Defined as schemas (name, description, parameter types) injected into the LLM's context so the model knows what actions are available. The tool layer handles registration, validation, execution, and result formatting.

Claude Code
Six categories: file operations, search, execution, web access, code intelligence, and subagent spawning.
OpenAI Agents SDK
Function tools (@function_tool), hosted tools (WebSearch, CodeInterpreter, FileSearch), and MCP server tools.
Schema Injection
Tool definitions go into context so the model knows what actions are possible at each turn.
Lazy Loading
Claude Code achieves 95% context reduction by exposing only the minimum tool set needed for the current step.

Memory — Multiple Timescales

Short-term memory is conversation history within a single session. Long-term memory persists across sessions.

Short-term Conversation history in the current session — lives in the context window.
Anthropic CLAUDE.md project files + auto-generated MEMORY.md. Three-tier hierarchy: lightweight index (~150 chars/entry, always loaded), detailed topic files on demand, raw transcripts via search only.
LangGraph Namespace-organized JSON Stores for cross-session persistence.
OpenAI Sessions backed by SQLite or Redis for durable state storage.

Critical design principle: the agent treats its own memory as a "hint" and verifies against actual state before acting. Memory is a claim about what existed when it was written — not necessarily what exists now.

Context Management — Where Agents Fail Silently

Context rot: model performance degrades 30%+ when key content falls in mid-window positions (Stanford's "Lost in the Middle"). Even million-token windows suffer instruction-following degradation as context grows.

Compaction Summarizes history approaching limits. Claude Code preserves architectural decisions and unresolved bugs while discarding redundant tool outputs.
Observation Masking JetBrains' Junie hides old tool outputs while keeping tool calls visible — reducing noise without losing the action trail.
Just-in-Time Retrieval Lightweight identifiers maintained; data loaded dynamically. Claude Code uses grep, glob, head, tail rather than loading full files.
Sub-agent Delegation Subagents explore extensively but return only 1,000–2,000 token condensed summaries to the parent context.
RESEARCH

ACON research: 26–54% token reduction while preserving 95%+ accuracy by prioritizing reasoning traces over raw tool outputs.

Prompt Construction — What the Model Actually Sees

Hierarchically assembles everything at each step. The ordering and priority of elements directly determines model behavior.

① System Prompt Highest priority. Server-controlled in OpenAI's Codex. Sets the agent's persona, capabilities, and core instructions.
② Tool Definitions Schema of all available tools injected so the model knows what actions are possible at this turn.
③ Memory Files Relevant long-term memory pulled in on demand: project context, user preferences, prior decisions.
④ Conversation History Prior turns in this session. Actively managed to avoid context rot.
⑤ User Message The current request. OpenAI Codex supports cascading AGENTS.md files (32 KiB limit) at this layer — developer instructions with decreasing priority.

Output Parsing — Structured Over Free-Text

Modern harnesses rely on native tool calling: the model returns structured tool_calls objects rather than free-text that must be regex-parsed.

Modern: Native Tool Calling

Model returns structured tool_calls. Harness checks: tool calls present? Execute and loop. None? Final answer.

Both OpenAI and LangChain support schema-constrained responses via Pydantic models.

Legacy: RetryWithErrorOutputParser

Feeds the original prompt, the failed completion, and the parsing error back to the model for a retry.

Still available for edge cases where structured output fails or models lack native tool support.

State Management — Persistence and Resume

How the harness checkpoints work, enables recovery after interruptions, and maintains continuity across context windows.

LangGraph Typed dictionaries flowing through graph nodes, reducers merging updates. Checkpointing at super-step boundaries enables resume after interruptions and time-travel debugging.
OpenAI (4 strategies) Application memory, SDK sessions, server-side Conversations API, or lightweight previous_response_id chaining. Mutually exclusive by design.
Claude Code Git commits as checkpoints + progress files as structured scratchpads. The filesystem itself provides continuity across context windows.

Error Handling — Errors Compound Fast

A 10-step process with 99% per-step success rate still has only ~90.4% end-to-end success. Error recovery is not optional in production.

Transient
Retry with exponential backoff. Network timeouts, rate limits, temporary failures. Stripe caps retries at two.
LLM-Recoverable
Return error as ToolMessage so the model can adjust its approach and retry differently — keeps the loop running.
User-Fixable
Interrupt for human input. The agent cannot self-correct — it needs information only the user can provide.
Unexpected
Bubble up for debugging. Do not silently swallow errors that indicate harness bugs — they will recur.
ANTHROPIC APPROACH

Anthropic catches failures within tool handlers and returns them as error results to keep the loop running — the model then decides how to adapt. Stripe's production harness caps retry attempts at two to prevent runaway loops.

Guardrails and Safety — Defense in Depth

OpenAI SDK — 3 Levels

Input guardrails: run on first agent before the model sees the message.

Output guardrails: run on final output before delivery to the user.

Tool guardrails: run on every tool invocation. A "tripwire" mechanism halts the agent immediately when triggered.

Anthropic Architecture

Permission enforcement is architecturally separate from model reasoning. The model decides what to attempt; the tool system decides what is allowed.

Claude Code gates ~40 discrete tool capabilities across three stages: trust establishment, permission check before each call, and explicit user confirmation for high-risk operations.

Verification Loops — What Separates Demos from Production

Rules-Based Tests, linters, type checkers. Deterministic ground truth — catches objective correctness failures before they propagate.
Visual Feedback Screenshots via Playwright for UI tasks. Captures what the model cannot infer from source code alone.
LLM-as-Judge A separate subagent evaluates output. Catches semantic issues but adds latency. Inferential, not deterministic.

Boris Cherny, creator of Claude Code: giving the model a way to verify its own work improves quality by 2–3×. Verification is not overhead — it is the mechanism that makes agents reliable.

Subagent Orchestration — Parallel Execution Models

Claude Code: Fork Byte-identical copy of parent context. Subagent inherits everything the parent knows at the moment of forking.
Claude Code: Teammate Separate terminal pane with file-based mailbox communication. Truly independent — no shared context.
Claude Code: Worktree Own git worktree and isolated branch per agent. Used for parallel development tasks that must not collide.
OpenAI: Agents-as-Tools Specialist handles a bounded subtask and returns result to the parent agent.
OpenAI: Handoffs Specialist takes full control from the parent — transfer of authority, not just data.
LangGraph Implements subagents as nested state graphs — composable and inspectable.

The Loop in Motion: A Step-by-Step Walkthrough

How all twelve components work together in a single TAO cycle. Step through each phase:

① Prompt Assembly
② LLM Inference
③ Output Classification
④ Tool Execution
⑤ Result Packaging
⑥ Context Update
⑦ Loop / Terminate
STEP 1

The harness constructs the full input: system prompt + tool schemas + memory files + conversation history + current user message. Critical context is positioned at the beginning and end of the prompt to counter the "Lost in the Middle" degradation.

STEP 2

The assembled prompt goes to the model API. The model generates output tokens — text, tool call requests, or both. The harness waits for the complete response before proceeding.

STEP 3

Three outcomes: (1) text with no tool calls → loop ends, this is the final answer; (2) tool calls present → proceed to execution; (3) handoff requested → update the current agent and restart the loop.

STEP 4

For each tool call: validate arguments, check permissions, execute in a sandboxed environment, capture results. Read-only operations can run concurrently; mutating operations run serially to prevent conflicts.

STEP 5

Tool results are formatted as LLM-readable messages. Errors are caught inside tool handlers and returned as error results — not exceptions — so the model can self-correct in the next turn rather than silently failing.

STEP 6

Results are appended to conversation history. If approaching the context window limit, the harness triggers compaction — summarizing history and discarding low-signal content while preserving key decisions and unresolved bugs.

STEP 7

Return to Step 1. A simple question: 1–2 turns. A complex refactoring task: dozens of tool calls across many turns. Termination conditions (layered): no tool calls produced · max turns exceeded · token budget exhausted · guardrail tripwire fires · user interrupts · safety refusal returned.

Step 1 of 7
LONG-RUNNING: THE RALPH LOOP

For tasks spanning multiple context windows, Anthropic developed a two-phase pattern. An Initializer Agent sets up the environment: init script, progress file, feature list, initial git commit. A Coding Agent in every subsequent session reads git logs and progress files to orient itself, picks the highest-priority incomplete feature, works on it, commits, and writes summaries. The filesystem provides continuity — not the context window.

How Real Frameworks Implement the Pattern FRAMEWORK-SPECIFIC

The same twelve components manifest differently across frameworks. Select a framework to see its architectural approach.

Framework: Anthropic OpenAI LangGraph CrewAI AutoGen

Anthropic — Claude Agent SDK

Exposes the harness through a single query() function that creates the agentic loop and returns an async iterator streaming messages. Philosophy: "dumb loop" — all intelligence lives in the model.

# Single entry point — harness manages everything below
async for message in query(prompt="refactor auth module", tools=tools):
    print(message)
Gather-Act-Verify Claude Code's cycle: gather context (search, read files) → take action (edit, run commands) → verify results (run tests, check output) → repeat.
Harness Thickness Bets on thin harnesses + model improvement. Anthropic regularly deletes planning steps from Claude Code as new model versions internalize those capabilities.

OpenAI — Agents SDK

Implements the harness through the Runner class with three modes. "Code-first": workflow logic expressed in native Python rather than graph DSLs.

result = await Runner.run(agent, "analyze this codebase")    # async
result = Runner.run_sync(agent, "analyze this codebase")   # sync
async for event in Runner.run_streamed(agent, msg):         # streamed
    handle(event)
Codex Architecture Three layers: Codex Core (agent code + runtime), App Server (bidirectional JSON-RPC API), client surfaces (CLI, VS Code, web). All surfaces share the same harness.
Priority Stack Server system message → tool definitions → developer instructions → user instructions (cascading AGENTS.md, 32 KiB limit) → conversation history.

LangGraph — Explicit State Graph

Models the harness as an explicit state graph. Two core nodes connected by a conditional edge. Evolved from LangChain's AgentExecutor, deprecated in v0.2 for being hard to extend and lacking multi-agent support.

graph.add_node("llm_call",  llm_call)
graph.add_node("tool_node", tool_node)
graph.add_conditional_edges(
    "llm_call",
    has_tool_calls,
    {True: "tool_node", False: END}
)
DEEP AGENTS

LangChain's Deep Agents explicitly use the term "agent harness": built-in tools, planning via a write_todos tool, file systems for context management, subagent spawning, and persistent memory. This is where the term entered mainstream practitioner vocabulary.

CrewAI — Role-Based Multi-Agent

Three core primitives define the architecture. The Flows layer adds a deterministic backbone on top of autonomous crew execution.

Agent
The harness around the LLM, defined by role, goal, backstory, and tools. The "who" in the system.
Task
The unit of work. Assigned to an agent with defined expected output and acceptance criteria.
Crew
The collection of agents. Orchestrates collaborative execution across the team toward a shared goal.
Flows Layer
"Deterministic backbone with intelligence where it matters" — manages routing and validation while Crews handle autonomous collaboration.

AutoGen / Microsoft Agent Framework — Conversation-Driven

Pioneered conversation-driven orchestration. Evolving into Microsoft Agent Framework. Three-layer architecture (Core, AgentChat, Extensions) supporting five orchestration patterns.

Sequential Agents execute in order. Simple pipelines where each step depends on the previous result.
Concurrent Fan-out / fan-in. Multiple agents work in parallel; results are consolidated by an aggregator.
Group Chat Multiple agents in a shared conversation, each contributing domain expertise.
Handoff Control transfers from one agent to another based on task domain or capability match.
Magentic A manager agent maintains a dynamic task ledger, coordinating specialists. The most sophisticated pattern — and the most expensive in LLM calls.

The Scaffolding Metaphor UNIVERSAL

METAPHOR

"Construction scaffolding is temporary infrastructure that enables workers to build a structure they couldn't reach otherwise. It doesn't do the construction. But without it, workers can't reach the upper floors."

The Co-Evolution Principle

Models are now post-trained with specific harnesses in the loop. Claude Code's model learned to use the specific harness it was trained with.

Changing tool implementations can degrade performance because of this tight coupling — the model has implicit expectations about tool behavior baked into its weights.

Manus: 5 Rebuilds in 6 Months

Each rewrite removed complexity: complex tool definitions became general shell execution; "management agents" became simple structured handoffs.

This is the trajectory. As models improve, harness complexity decreases. The scaffolding is removed when the building is complete.

The "future-proofing test" for harness design: if performance scales up with more powerful models without adding harness complexity, the design is sound. If you need more harness complexity to get better results from a better model, something is wrong with the architecture.

Seven Decisions That Define Every Harness UNIVERSAL

Every harness architect faces these choices. Each involves a genuine tradeoff — there is no universally correct answer.

Single vs. Multi-Agent Both Anthropic and OpenAI: maximize a single agent first. Multi-agent adds overhead — extra LLM calls for routing, context loss at handoffs. Split only when tool overload exceeds ~10 overlapping tools, or clearly separate task domains exist.
ReAct vs. Plan-Execute ReAct interleaves reasoning and action at every step — flexible but higher per-step cost. Plan-and-execute separates planning from execution. LLMCompiler reports 3.6× speedup over sequential ReAct for parallelizable tasks.
Context Window Strategy Five production approaches: time-based clearing · conversation summarization · observation masking · structured note-taking · sub-agent delegation. ACON: 26–54% token reduction while preserving 95%+ accuracy by prioritizing reasoning traces over raw tool outputs.
Verification Loop Design Computational verification (tests, linters) provides deterministic ground truth. Inferential verification (LLM-as-judge) catches semantic issues but adds latency. Thoughtworks frames this as guides (feedforward, steer before action) vs. sensors (feedback, observe after action).
Permission Architecture Permissive (fast but risky, auto-approve most actions) vs. restrictive (safe but slow, require approval for each action). The correct choice depends entirely on deployment context and the consequences of an error.
Tool Scoping More tools often means worse performance. Vercel removed 80% of tools from v0 and got better results. Claude Code achieves 95% context reduction via lazy loading. Expose the minimum tool set needed for the current step — not everything the agent could ever need.
Harness Thickness How much logic lives in the harness vs. the model. Anthropic bets on thin harnesses + model improvement. Graph-based frameworks bet on explicit control. No universal right answer — depends on which direction you expect to get leverage from.

WHERE THE HARD ENGINEERING LIVES

Managing context as a scarce resource. Designing verification loops that catch failures before they compound. Building memory systems that provide continuity without hallucination. Making architectural bets about how much scaffolding to build versus how much to leave to the model. The harness is not a solved problem or a commodity layer.

The field is moving toward thinner harnesses as models improve. But the harness itself is not going away. Even the most capable model needs something to manage its context window, execute its tool calls, persist its state, and verify its work. The next time your agent fails, don't blame the model. Look at the harness.
