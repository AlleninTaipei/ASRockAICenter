The Token Problem

Why your AI assistant burns money on garbage — and how RTK stops it

Imagine you ask a friend to help you fix a bug. Before answering, they read every line of your 2,000-page engineering manual out loud — every time you ask a question. That's what happens when you run a CLI command in Claude Code without RTK.

💰

Without RTK

git log -20 sends 2,500 tokens to the AI — full commit hashes, author emails, timestamps, even blank lines.

✂️

With RTK

Same command sends just 500 tokens — the 5 most important pieces: hash, message, author, and time. Same information, 80% less cost.

📊

In a 30-min session

That difference adds up. RTK saves roughly ~94,000 tokens per session — about an 80% reduction in what you pay for.

First: what's a token?

Think of tokens like the minutes on an old phone plan. Your LLM charges you for every word it reads — including all the noise your terminal commands spit out.

What lives inside each module

Every command module in RTK follows the same blueprint — like a standardized franchise location. Walk into any one and you'll recognize the layout.

src/cmds/git/git.rs 

pub enum GitCommand {
    Diff,
    Log,
    Status,
    Commit,
    Push,
    Pull,
}
PLAIN ENGLISH

The git module defines a list of sub-commands it knows how to handle...

"diff" = show what changed in files

"log" = show commit history

"status" = show what files have changed

"commit" = save a snapshot

"push" = upload to GitHub

"pull" = download from GitHub

Each one gets a different filter strategy.

📌
Enum: the menu of options

An enum in Rust is like a dropdown menu with fixed choices. The git module's enum says: "I know about these sub-commands, and nothing else goes through me."

The secret weapon: lazy_static regex

Every module uses pattern-matching to decide what to trim. The patterns are pre-compiled once at startup — like a factory calibrating its cutting machines once before a production run, not before every piece.

src/cmds/git/git.rs — pattern matching 

lazy_static! {
    static ref COMMIT_RE: Regex =
        Regex::new(r"^[0-9a-f]{7,40}")
            .unwrap();
}
PLAIN ENGLISH

Create a pattern that gets compiled once and reused forever...

"static ref" = shared globally, computed on first use

COMMIT_RE = a regex that matches commit hashes

The pattern: starts with hex chars, 7-40 long

.unwrap() = "crash immediately if pattern is wrong" (only safe here because it's a fixed literal)

⚡

Why compile once?

Compiling a regex is slow. With 100 commands running per session, compiling on every call would add 50ms+ overhead. Once is free.

🏎️

RTK's startup target

Every RTK command must add less than 10ms overhead. Most add 5-8ms. This is possible because all the "expensive" work happens only once.

Phase 1 + 2: Parse and Route

When you type rtk git log -10, a library called Clap breaks it apart like a maitre d' reading a reservation slip — who's the guest, what table, any special requests?

src/main.rs — command routing 

match cli.command {
    Commands::Git { args, .. } => {
        git::run(args, verbose)
    }
    Commands::Cargo { args, .. } => {
        cargo_cmd::run(args, verbose)
    }
    Commands::Npm { args, .. } => {
        npm_cmd::run(args, verbose)
    }
}
PLAIN ENGLISH

Look at the first word after "rtk" and decide what to do...

If it's "git", send to the git module with the remaining args

Run the git module's entry function

(close the git case)

If it's "cargo", route to the cargo module instead

(each command family has its own handler)

If it's "npm", route to the npm module

(the pattern repeats for all 100+ commands)

Phase 3 + 4: Execute and Filter

The real command runs — RTK doesn't pretend to be git, it calls git. Then the output goes through the filter like water through a sieve: noise falls out, signal stays in.

Fallback pattern — src/cmds/git/git.rs 

let filtered = filter_output(&output.stdout)
    .unwrap_or_else(|e| {
        eprintln!("rtk: filter warning: {}", e);
        output.stdout.clone()
    });
PLAIN ENGLISH

Try to run the filter on the command's output...

If filtering fails for any reason, run this fallback...

Print a warning to stderr (not to Claude's input)

Return the original, unfiltered output instead

(Claude gets full output — never nothing)

🛡️
The Fail-Safe Rule

RTK never blocks you. If its filter crashes or produces bad output, it falls back to the raw original. This is called the fallback pattern — it's why you can trust RTK in production without fear.

📝

stderr vs stdout

stderr is for warnings; stdout is for results. RTK's warnings go to stderr so they never appear in Claude's context window.

🚦

Exit codes

If cargo test exits with code 1 (failure), RTK propagates that exact code. CI/CD pipelines depend on this — a green proxy hiding a red test would be catastrophic.

Phase 5 + 6: Print and Track

After filtering, the compressed output goes straight to stdout for Claude to read. Simultaneously, RTK records the savings in a local SQLite database.

src/core/tracking.rs — recording savings 

pub struct TimedExecution {
    start: Instant,
}

impl TimedExecution {
    pub fn start() -> Self {
        Self { start: Instant::now() }
    }
}
PLAIN ENGLISH

A small helper for timing how long a command takes...

It stores the moment it was created (the start time)

(end of the struct definition)

(implementation block — adding methods to the struct)

The "start a timer" function — returns a new TimedExecution

Record "right now" as the start time

(close the function, close the impl block)

🗄️
Why SQLite?

RTK needs to store historical data (every command you run) without requiring you to install a database server. SQLite is a database in a single file — stored at ~/.local/share/rtk/tracking.db. It's the same database that powers browsers, iOS apps, and Android phones.

The Hook System

How RTK plugs invisibly into Claude Code, Cursor, and other AI editors

Running rtk git log manually works — but what if you forget? The hook system makes RTK automatic: it intercepts commands before the AI sees them, like a customs officer screening luggage at an airport border, automatically.

✈️

Without hooks

You have to remember to type rtk before every command. Forget once, and Claude reads 2,500 tokens of raw git output.

🛃

With hooks

Claude Code intercepts every command and rewrites git log to rtk git log automatically — 100% adoption, zero mental overhead.

🔌
Supported AI tools

RTK hooks work with: Claude Code, Cursor, Windsurf, Cline/Roo Code, Kilo Code, and others. Run rtk init to install the hook for your tool automatically.

Two strategies: Rewrite vs. Suggest

RTK offers two hook modes — like the difference between a GPS that automatically reroutes you versus one that suggests a better route but lets you decide.

Installing and trusting the hook

Running rtk init writes the hook script into your AI tool's config directory. RTK also verifies the hook hasn't been tampered with — like a pharmacy checking a prescription's seal before dispensing.

src/hooks/integrity.rs — SHA-256 verification 

// Verify hook hasn't been modified
// Uses SHA-256 hash of the hook content
// to detect unauthorized changes
PLAIN ENGLISH

Before trusting the hook script...

Compute a SHA-256 fingerprint of its contents

Compare against the expected fingerprint

If they differ, refuse to run — something's wrong

🔐

Why verify integrity?

A hook that runs before every command is a high-value target. If someone modified it to inject malicious commands, you'd never know. SHA-256 verification catches any change immediately.

🌐

Trust per project

RTK tracks which projects you've explicitly trusted. Hooks only run in trusted project directories — protecting you from running malicious code in unfamiliar repos.

The Analytics Brain

How RTK tracks every token it saves — and shows you the evidence

After every command, RTK silently logs the before-and-after token counts to a local database. Then rtk gain reads that history and shows you exactly how much money and context your AI saved — like a car's trip odometer that tracks every mile automatically.

📈

rtk gain

Shows total tokens saved, savings percentage, and estimated dollar cost reduction for your session or project.

🕐

rtk gain --history

Shows a breakdown by command — which commands saved the most, which ran most often, trends over time.

🗂️

rtk session

Shows per-session breakdowns — how much was saved in today's Claude Code session vs. yesterday's.

Inside the tracker: what gets recorded

Every time RTK filters a command, it records a row in the SQLite database. Think of it like a receipt printer — quiet, automatic, always running in the background.

src/core/tracking.rs — what gets stored 

// Storage: SQLite database
// Path: ~/.local/share/rtk/tracking.db
// Retention: 90-day auto-cleanup
// Metrics recorded per command:
//   - input_tokens  (before filter)
//   - output_tokens (after filter)
//   - savings_pct   (% reduction)
//   - execution_ms  (RTK overhead)
//   - timestamp     (when it ran)
PLAIN ENGLISH

The data lives in a single file database...

At this path on your computer

Entries older than 90 days are deleted automatically

For each command, these numbers are saved:

How many tokens the raw output had

How many tokens RTK's filtered version has

The percentage saved (e.g., 80%)

How much time RTK itself added (usually 5-8ms)

The exact time the command ran

💡
How token counting works

RTK estimates token counts by dividing character count by 4 (a common approximation — LLMs average about 4 characters per token). It's not exact, but it's fast, offline, and consistent across sessions. Exact counting would require calling the AI — which defeats the purpose.

How rtk gain reads the data

The analytics layer lives in src/analytics/ — separate from the filter modules. It reads the database, aggregates the numbers, and formats them for display. Clean separation of concerns.

src/core/tracking.rs — Tracker API 

let tracker = Tracker::new().unwrap();
let summary = tracker.get_summary().unwrap();
println!("Saved {} tokens", summary.total_saved);
