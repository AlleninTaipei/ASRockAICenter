What Claw Does

An AI that lives in your terminal — and can actually do things

Imagine having a coding assistant that never sleeps

You type a sentence. It reads your files, runs commands, edits code, searches the internet — and writes back what it did. That's Claw Code: an open-source CLI that puts an AI coding agent right in your terminal.

It's not a chat window. It's a coding agent — meaning it can read files, modify code, and execute commands on your computer, on your behalf.

💬

Interactive REPL

Have a back-and-forth conversation. Ask questions, give instructions, and build on previous turns.

⚡

One-Shot Prompts

Run a single command non-interactively: claw prompt "summarize this repo"

🤖

Multi-Provider AI

Works with Claude, GPT, Grok, Qwen, or even local models running on your own machine.

Imagine you type: claw prompt "say hello"

Here's everything that happens in the next second — from your keyboard to the AI and back.

1
The CLI starts up

The claw binary launches, loads your config, checks your API key, and assembles a "system prompt" that tells the AI what tools it has available.

2
Your message is packaged

Your text ("say hello") is wrapped into a structured message object with a role ("user") and sent to the AI over the internet via HTTPS.

3
The AI responds in a stream

Rather than waiting for the whole answer, words arrive one-by-one like a live typing animation. The AI's response "streams" in as it's generated.

4
Rendered to your terminal

The response is formatted — markdown converted to readable text, code blocks highlighted — and printed to your screen.

5
Session saved

The whole conversation is written to a JSONL file so you can resume it later with --resume latest.

💡
Why Rust?

Claw Code is written in Rust — a language that compiles to a single fast binary with no runtime overhead. That means faster startup, no hidden crashes from memory bugs, and a single file you can ship anywhere. When you run cargo build, you get one claw executable that needs nothing else installed.

The commands that matter most

Once you have it built, here are the three entry points you will use every day:

claw Launch the interactive REPL — type your message, press Enter, see the response. Works like a chat app in your terminal.
claw prompt "…" One-shot mode — pass a single instruction and exit when done. Great for scripts and automation.
claw --resume latest Pick up exactly where you left off — all previous context is restored from the saved session file.
/doctor The built-in health check. Run this first after installing — it verifies your API key, permissions, and configuration are all set up correctly.
🔑
One prerequisite: an API key

Claw connects to AI models over the internet using an API key. You need export ANTHROPIC_API_KEY="sk-ant-..." in your terminal before running claw. A Claude subscription alone is NOT enough — you need a separate API key from console.anthropic.com.

The Cast of Characters

Eight Rust crates, each with a distinct job — meet the team behind every claw session

Claw is not one program — it is eight

Think of it like a theatre production. The audience sees one show, but backstage there are actors, lighting engineers, sound crew, a director, and a stage manager — each with a specific role. Claw works the same way: eight crates working together, each owning one responsibility.

rust/crates/ The Rust workspace — all eight crates live here
rusty-claude-cli/ 🎭 The director — the main binary you actually run
runtime/ 🧠 The brain — conversation loop, sessions, hooks, MCP
api/ 📡 The radio — talks to Anthropic, OpenAI, xAI over the internet
tools/ 🔧 The hands — executes file reads, bash commands, searches
plugins/ 🔌 The extension board — loads custom hooks and plugin scripts
commands/ 📋 The manual — handles slash commands like /help and /status
telemetry/ 📊 The ledger — tracks token usage and cost
mock-anthropic-service/ 🧪 The rehearsal stage — a fake API server for testing

The three crates that do most of the work

🎭
rusty-claude-cli — The Front Door

This is the claw binary. It handles everything you see: parsing your command-line flags, drawing the spinner while the AI thinks, streaming words to your terminal as they arrive, and wiring all the other crates together. It is the only crate that knows about your screen.

🧠
runtime — The Conversation Brain

This crate owns the logic of each turn: building the message payload, deciding what tools the AI can use, enforcing permissions, saving the session to disk, running hooks before and after tool calls, and compacting long conversations so they fit in the AI's memory window.

📡
api — The Network Layer

This crate speaks to AI providers. It knows how to format a request for Anthropic's API vs. OpenAI's API, how to parse the streaming response, and how to implement prompt caching (a trick that saves money on repeated requests).

How the api crate handles multiple AI providers

The api crate uses a Rust pattern called an enum to represent whichever AI provider is active. Here is the real code:

CODE — api/src/client.rs 
pub enum ProviderClient {
  Anthropic(AnthropicClient),
  Xai(OpenAiCompatClient),
  OpenAi(OpenAiCompatClient),
}

pub fn from_model(model: &str)
  -> Result<Self, ApiError> {
  let resolved = resolve_model_alias(model);
  match detect_provider_kind(&resolved) {
    Anthropic => Ok(Self::Anthropic(...)),
    Xai => Ok(Self::Xai(...)),
    OpenAi => Ok(Self::OpenAi(...)),
  }
}
PLAIN ENGLISH

A ProviderClient is always exactly one of three things: an Anthropic client, an xAI (Grok) client, or an OpenAI-compatible client.

Anthropic has its own client type because its API is different from OpenAI's.

xAI (Grok) uses the same wire format as OpenAI, so they share an implementation.

Same for OpenAI — one implementation serves many providers.

 

Given a model name like "gpt-4o" or "claude-opus-4-6", figure out which provider to use...

...first expand shorthand like "opus" → "claude-opus-4-6"...

...then look at the name to decide: "claude" → Anthropic, "grok" → xAI, anything else → OpenAI-compat.

💡
This is called a Strategy Pattern

The same function call (stream(request)) works regardless of which provider is active underneath. The enum hides the difference. This is one of the most common design patterns in software — give different implementations the same "interface" so the rest of the code does not have to know which one it is using.

Why the response appears word by word

When you ask an AI a question, it does not generate the entire answer and then send it all at once. It generates tokens one at a time — and Claw displays each token the moment it arrives.

🎬
Think of it like a live sports ticker

A sports ticker does not wait for the game to end before reporting the score. It streams live updates as they happen. SSE (Server-Sent Events) works the same way — the connection stays open and data flows through it continuously. Claw uses an SSE decoder to parse each incoming chunk and display it immediately.

CODE — runtime/src/conversation.rs 
pub enum AssistantEvent {
  TextDelta(String),
  ToolUse {
    id: String,
    name: String,
    input: String,
  },
  Usage(TokenUsage),
  MessageStop,
}
PLAIN ENGLISH

An AssistantEvent is one of four types of things that can arrive from the AI stream.

TextDelta: a small chunk of text (could be one word, half a word, or just a space). Render it immediately.

ToolUse: the AI wants to call a tool. It includes...

...a unique ID (for tracking which tool call this is)...

...the tool name (e.g. "read_file")...

...and the arguments (e.g. which file path to read).

Usage: token counts — how many tokens were used (for billing and cost tracking).

MessageStop: the AI has finished generating. Clean up and wait for the next turn.

What happens when conversations get too long

AI models have a context window — a maximum amount of text they can hold in memory. After a long session, claw automatically "compacts" the conversation.

1
Estimate token usage

Runtime counts how many tokens the current session is using. When it exceeds ~100,000 tokens, it kicks off compaction.

2
Ask the AI to summarize

The AI is asked to write a compact summary of what has happened so far. Only the last 4 messages are kept verbatim.

3
Replace history with summary

All the old messages are swapped out for the summary. The session continues with full context but a much smaller footprint.

💡
Why this matters for you as an AI director

When you notice claw say "this session is being continued from a previous conversation," compaction has fired. If you are mid-task, the AI may lose detail about early decisions. You can prevent this by starting new sessions for large, independent tasks — or by being explicit in your instructions about what context matters.

Tools & the Permission System

How the AI safely does real work on your machine — and who decides what it is allowed to do

The AI can read your files. Should it also delete them?

Claw gives the AI real tools — it can read files, write files, run bash commands, search your codebase, and make network requests. That is enormously powerful. It is also why there is a permission system between the AI's requests and your machine.

Think of it like an apartment building with a key card system. The AI is a visiting contractor: it can enter rooms it is authorized for, but the building manager (the permission system) controls which doors its card opens.

read-only The AI can read files and search the codebase, but cannot write, delete, or run shell commands. Safe for exploration.
workspace-write The AI can read and write files within the current project directory. It can also run bash commands that do not affect anything outside the workspace.
danger-full-access No restrictions. The AI can run any command, modify any file. Use only when you fully understand and trust the task being performed.
⚠️
The permission mode is not an AI setting — it is a hard boundary

Even if you ask the AI to "please delete everything," in read-only mode it physically cannot — the permission enforcer will block the tool call before it executes. The AI's intent does not override the mode.

What tools does the AI actually have?

The tools crate provides a set of built-in capabilities that the AI can request. These are declared upfront in the system prompt so the AI knows what it can do before your first message.

📄

read_file

Read the contents of any file. Supports offset and line limits so it can read specific sections of large files.

✏️

write_file / edit_file

Create a new file or apply a precise diff to an existing one. The AI sends exact old/new strings — no guessing.

🔎

glob_search / grep_search

Find files by pattern (e.g. **/*.rs) or search file contents by regex. Equivalent to find and grep.

💻

execute_bash

Run a shell command. Requires workspace-write or higher. The output (stdout + stderr) is returned to the AI.

🔌

MCP tools

MCP servers can expose additional tools — database queries, web browsing, custom APIs — following the same permission model.

How a tool call gets checked before it runs

Every tool invocation goes through the permission system. Here is the real data structure that represents a permission check:

CODE — runtime/src/permissions.rs 
pub enum PermissionMode {
  ReadOnly,
  WorkspaceWrite,
  DangerFullAccess,
  Prompt,
  Allow,
}

pub struct PermissionRequest {
  pub tool_name: String,
  pub input: String,
  pub current_mode: PermissionMode,
  pub required_mode: PermissionMode,
}
PLAIN ENGLISH

A session is always in one of these five authorization levels.

ReadOnly: look but do not touch.

WorkspaceWrite: read and write files in the project folder.

DangerFullAccess: no restrictions whatsoever.

Prompt: ask the user before each sensitive action.

Allow: a specific override to permit one particular action.

 

When a tool call arrives, this bundle is assembled to evaluate it.

Which tool is being requested? (e.g. "execute_bash")

What arguments? (e.g. the shell command to run)

What mode is the current session in?

What mode does this tool require? If current < required → blocked.

Hooks: run your own code before or after any tool

The plugins crate lets you register shell scripts that run automatically around tool calls. Think of it like airport security — you configure it once, and every tool call passes through your checkpoint.

⬅
PreToolUse hook

Fires before a tool executes. Your script can inspect the tool name and input, then output allow, deny, or ask to override the permission decision.

➡
PostToolUse hook

Fires after a tool completes successfully. Your script receives the tool output — useful for logging, auditing, or triggering side effects.

⚠
PostToolUseFailure hook

Fires when a tool fails. Lets you react to errors — send an alert, write to a log, or attempt recovery.

💡
Hooks are how you build guardrails without modifying claw itself

Want to prevent the AI from ever touching *.env files? Write a PreToolUse hook that inspects the input and outputs deny when it sees that pattern. Want to post a Slack notification every time the AI commits code? Add a PostToolUse hook on execute_bash. This is where power users customize claw for their team.

The Big Picture

Multi-provider AI, autonomous agents, and what it all means for how software gets built

Claw is not married to Claude

One of the most strategically important features of claw is that it works with any major AI provider — not just Anthropic. This matters because models are evolving fast, prices change, and the best model for one task may not be the best for another.

🟠
Anthropic (Claude)

The default provider. Uses the native Anthropic Messages API with streaming and prompt caching. Set ANTHROPIC_API_KEY.

⚡
xAI (Grok)

Elon Musk's AI lab. Uses the OpenAI-compatible wire format. Set XAI_API_KEY and use --model grok.

🔵
OpenAI-compatible (GPT, OpenRouter, Ollama, Qwen…)

Anything that speaks the OpenAI Chat Completions format works. Point OPENAI_BASE_URL at any compatible service, including local models.

🦙
Local models via Ollama

Run a model entirely on your own machine — no API key, no internet, no cost per token. Set OPENAI_BASE_URL=http://localhost:11434/v1 and use any Ollama model name.

🗺️
How claw picks the right provider automatically

Model name prefixes act as routing signals. claude-* → Anthropic. grok-* → xAI. openai/* or gpt-* → OpenAI-compat. qwen-* → Alibaba DashScope. If none match, it falls back to whichever API key is set in your environment. You rarely need to specify the provider explicitly.

Claw is not just a tool — it is a demonstration

The repository that contains claw was itself built by AI agents. Not assisted — actually built: agents planning, coding, testing, reviewing, and pushing, while humans provided direction from a Discord channel. The code is evidence; the process is the product lesson.

🦞

clawhip

The event router. Watches git commits, tmux sessions, GitHub events, and agent lifecycle — and delivers notifications outside the coding agent's context window.

🤝

oh-my-openagent

The multi-agent coordinator. Manages planning, handoffs between Architect, Executor, and Reviewer agents, and convergence when they disagree.

🔁

oh-my-codex (OmX)

The workflow layer. Turns short directives into structured execution: planning keywords, parallel multi-agent workflows, and persistent verification loops.

What the bottleneck actually is now

When agent systems can rebuild a codebase in hours, the scarce resource is no longer typing speed. Understanding this changes how you should spend your time.

🧭
Direction

Knowing what to build — and why it matters — is more valuable than knowing how to build it. Agents are fast; bad direction is expensive to undo.

🔬
Decomposition

Breaking a big goal into well-scoped tasks that can be parallelized is a skill machines do not yet have. This is your leverage.

🎯
Taste

Recognizing when an AI-generated solution is correct but wrong — technically working but architecturally bad — requires judgment that only comes from understanding the system.

🔤
Vocabulary
