What Is n8n?

You built something. Now let us see how it works.

n8n is a workflow automation platform. You connect apps by dragging nodes onto a canvas — and when a trigger fires, data flows through those nodes automatically.

No code required — but understanding the internals means you can steer AI assistants precisely, debug issues fast, and make smart architectural decisions.

Nodes are specialists. Connections are the wires.

Think of a circuit board — each component (node) does one thing, and copper traces (connections) carry the signal from one to the next. n8n is your circuit board designer.

🔌
How Data Moves

Every node has an INPUT and an OUTPUT. Data arrives as a list of items, gets processed, then passes to the next node.

1
Trigger Node fires

Something happened in the world — a webhook arrived, a timer went off, someone clicked Execute.

2
First node receives the data items

The trigger node outputs its data as a list of items. Each item is a JSON object with fields.

3
Node transforms, filters, or enriches the data

Each subsequent node does its specific job — filter, rename, look up, call an API, run AI.

4
Output passes to the next node

The processed items flow forward along the connection wire — or the workflow ends at the last node.

What data actually looks like inside a node.

Every node receives and outputs data in the same shape: a array of items, where each item has a json field containing your data as key-value pairs.

CODE 

// What one item looks like
{
  json: {
    name: "Alice",
    email: "alice@example.com",
    score: 92
  }
}

// What the engine passes between nodes
const items = [item1, item2, item3];
const result = await node.execute(items);
PLAIN ENGLISH

One item is a wrapper around your data — the json field holds the actual key-value pairs you work with in expressions

Each data record from your trigger (a form response, a spreadsheet row, a webhook payload) becomes one item in the list

The engine hands the full list to the node and waits for it to return a new list

await means: pause here and wait for the node to finish before moving to the next one

Every workflow starts with a trigger.

The first node in every workflow is always a trigger node — it decides when the workflow runs. Pick the wrong trigger type and your automation will never fire (or fire at the wrong time).

🌐
Webhook Trigger

An external service calls n8n directly via HTTP. You give the third-party service your unique n8n webhook URL, and whenever that service has something to tell you, it sends a POST request.

⏰
Schedule Trigger

Runs automatically every N minutes, hours, or days — no external call needed. n8n uses a cron schedule to wake up the workflow at exactly the right time.

👆
Manual Trigger

You click Execute Workflow in the editor to test. Great during development — you control exactly when it runs and can inspect every node output immediately.

📱
App Event Trigger

A specific service fires when something happens — new email, new row, new ticket. Uses the service long-polling or push API to detect the event.

Five moving parts. One automation platform.

Where to find things in the codebase.

n8n is a monorepo managed with pnpm workspaces. When you ask an AI to write code for n8n, specifying the correct package is the difference between a working change and a dead end.

The Vue 3 frontend canvas — this is what you see in your browser. Node rendering, drag-and-drop, the sidebar, all UI text.

CLI
packages/cli/

The Express backend — REST API, webhooks, workflow runner. If you want to add a new API endpoint or change how executions are stored, this is your package.

CTL
packages/cli/src/controllers/

One file per API route group: auth, users, workflows, credentials, executions, license. Each file wires URL patterns to business logic functions.

EXE

Everything about running and storing workflow executions — the controller, repository, and service that manages execution lifecycle.

WF
packages/workflow/

Core types and the Workflow class — shared by both frontend and backend. When you see a TypeScript interface like IWorkflow or INode, it lives here.

ENG
packages/core/

The execution engine — runs nodes, manages data flow, handles credential resolution. This is the heart of n8n.

NOD
packages/nodes-base/

All 400+ built-in node integrations — one subfolder per integration. Each node implements the INodeType interface with a description and an execute() method.

03

How a Workflow Runs

From trigger to completion — the six-step sequence every n8n builder should know.

A trigger fires. What happens next?

From the moment a webhook arrives to the moment execution is complete, n8n runs through a precise 6-step sequence — and knowing this sequence is your superpower for debugging.

1
Trigger detected

A webhook, schedule, or manual click fires — the workflow is woken from sleep.

2
WorkflowRunner picks up the job

The WorkflowRunner validates the workflow and checks that all credentials are present.

3
Execution record created

A unique ID is assigned, status is set to "running," and the run is written to the database so you can inspect it later.

4
Trigger node runs first

The first node generates the initial data items — the raw payload from the webhook, schedule metadata, or whatever the trigger produced.

5
Each subsequent node runs in order

Each node receives the output items from the previous node, transforms them, and outputs new items — one workstation at a time along the assembly line.

6
Execution complete

Status is updated to "success" or "error," results are saved, and if triggered by a webhook, the HTTP response is sent back to the caller.

🏭
The Factory Metaphor

Think of a factory assembly line. A product (the data) enters at one end (the trigger), passes through each workstation (node) in sequence, gets transformed at each step, and exits at the other end. The Workflow Engine is the factory floor manager deciding what order things happen.

Data always travels as a list of items.

Imagine a spreadsheet: each row is one item, with columns for JSON data (the content) and binary data (files). Every node receives rows, processes them, and returns new rows.

CODE 

export interface INodeExecutionData {
  json: IDataObject;
  binary?: IBinaryKeyData;
  pairedItem?: IPairedItemData | IPairedItemData[] | number;
  error?: NodeApiError | NodeOperationError;
  index?: number;
}
PLAIN ENGLISH

Every item has a json field — this is the actual data: an object with key-value pairs like { name: "Alice", score: 42 }.

Optionally, an item can carry binary data (files, images, PDFs) alongside the JSON — the ? means this field is not always present.

pairedItem links an output item back to which input item it came from — useful when merging two branches or debugging where data originated.

If this item represents an error, the error field holds details about what went wrong — so errors travel as items too, not as crashes.

index tracks this item's position when processing — used for batch operations and loop tracking.

💡
When You Write {{ $json.name }}

You are reading from the json field of the current item. When AI writes expressions for you, it is working with this exact structure — the INodeExecutionData interface is the contract that every node in n8n must follow.

How the engine decides what to run next.

The Workflow Engine uses a stack (like a to-do list) to track which nodes still need to run. Here is the real code from the n8n codebase.

CODE 

const nodeExecutionStack: IExecuteData[] = [
  {
    node,
    data: {
      main: data,
    },
    source: null,
  },
];

const executionData = createRunExecutionData({
  executionData: {
    nodeExecutionStack,
  },
});
PLAIN ENGLISH

Create a list of nodes waiting to run — initially contains just the starting (trigger) node.

node — the node that should run next (the trigger to start).

data.main — the data to feed into that node. "main" is the primary input connection type in n8n.

source: null — no parent node. This is the start of the chain; nothing fed data to the trigger.

Wrap the stack in the full execution data structure that the engine reads throughout the run.

The engine will keep pulling nodes off this stack, running them, and pushing their connected children back on — until the stack is empty.

🔍
Why This Matters for Debugging

When you tell an AI assistant "node X never ran," you are saying it was never added to the execution stack. The most common reason: there was no data item coming from its parent node — an empty list stops the chain.

Nodes: The 400+ Building Blocks

Every node is a specialist. Learn to speak their language and AI can build anything for you.

Every node is a specialist with one job.

Whether it is Gmail, Slack, Google Sheets, or a custom HTTP call — every node in n8n follows the same contract: receive items, process them, return items.

⚡

Trigger Nodes

Start the workflow. Fire when something happens — a webhook, a schedule, a new email, a new row. Always the first node.

🔧

Action Nodes

Do something: send a message, create a record, call an API. The workhorses of automation.

🔀

Transform Nodes

Reshape data: filter items, merge branches, set fields, split lists. No API call — pure data manipulation. The Set, Filter, and Merge nodes are in this family.

🤖

AI / Logic Nodes

LangChain-powered: chat with AI models, run agents, classify text, generate content. Found in the AI section of the node panel.

What a node looks like under the hood.

Every node is a TypeScript class with a description (its configuration) and an execute() method. Here is the real HttpRequest node from the n8n codebase.

CODE 

export class HttpRequest extends VersionedNodeType {
  constructor() {
    const baseDescription: INodeTypeBaseDescription = {
      displayName: 'HTTP Request',
      name: 'httpRequest',
      icon: 'node:http-request',
      group: ['output'],
      description: 'Makes an HTTP request and returns the response data',
      defaultVersion: 4.4,
    };

    const nodeVersions = {
      1: new HttpRequestV1(baseDescription),
      4.4: new HttpRequestV3(baseDescription),
    };

    super(nodeVersions, baseDescription);
  }
}
PLAIN ENGLISH

Define a new node class called HttpRequest — a versioned node that can have multiple implementations.

displayName is what appears in the node search panel when you type "HTTP".

name is the internal ID stored in workflow JSON files — never shown to users, but referenced in code.

This node belongs to the output group — meaning it sends data out to external services.

The human-readable description shown as a subtitle in the node editor.

The current default version is 4.4 — new workflows use this version automatically.

Map version numbers to actual implementation classes — version 1 uses the old code, version 4.4 uses the modern rewrite.

Old workflows keep working because they pin to the version they were built with — upgrades never break existing automations.

💡
Versioned Nodes: Your Safety Net

Versioned nodes are why you can safely upgrade n8n without breaking existing workflows. Old workflows continue to use the exact version they were built with — the new version only activates for newly created nodes.

Credentials: your keys, encrypted and safe.

A credential is like a key card in a hotel — you configure it once at the front desk (n8n settings), and every node that needs access uses the same card without you having to hand over the actual key each time.

1
You add a credential

You paste a Slack API token into n8n's credential form and click Save.

2
n8n encrypts it immediately

The token is encrypted using the server-side encryption key before anything is written to disk.

3
The encrypted blob is stored

The database stores only the scrambled version — never the raw token. This is called encryption at rest.

4
Decrypted in memory at execution time

When a workflow runs, the engine decrypts the credential in memory for that one execution only.

5
Node uses it — then it is gone

The node receives the decrypted value, makes the API call, and the plaintext is discarded. It never touches the filesystem.

🔒
Why You Cannot See Your Token After Saving

The value is encrypted at rest. If someone steals your database file, they cannot read your credentials without the encryption key — which lives separately on the server. This is intentional, not a bug.

Expressions & AI

The {{ }} that makes every workflow dynamic — and the AI nodes that let your automations think.

That {{ }} is not decoration — it is code running live.

Expressions let you make node parameters dynamic — instead of typing a fixed value, you write a tiny program inside double curly braces that runs at execution time. The result replaces the expression, giving each item its own personalized value.

Think of it like a mail merge in Microsoft Word — instead of typing each person's name into 1000 letters, you write {{name}} and the system fills it in for each recipient. n8n expressions work the same way: {{ $json.email }} gets replaced at runtime with the actual email from each data item.

📦

$json

The current item's JSON data. {{ $json.email }} reads the 'email' field from the current item being processed.

🔗

$node

Data from any previous node by name. {{ $node['HTTP Request'].json.status }} reaches back to an earlier node in the workflow regardless of how many steps away it is.

⚙️

$workflow

Workflow-level info. {{ $workflow.id }} and {{ $workflow.name }} give you metadata about the workflow itself.

💡
Any JavaScript Is Valid

You can write any JavaScript inside {{ }}. Math, string manipulation, date formatting — it is all valid. {{ new Date().toISOString() }} gives you the current timestamp in ISO format.

How expressions actually get evaluated.

When a node runs and encounters {{ $json.name }}, the expression engine kicks in. Here is what happens behind the scenes in packages/workflow/src/expression.ts.

CODE 

const IS_FRONTEND = typeof process === 'undefined' || IS_FRONTEND_IN_DEV_MODE;

const isSyntaxError = (error: unknown): error is SyntaxError =>
  error instanceof SyntaxError || (error instanceof Error && error.name === 'SyntaxError');

const isExpressionError = (error: unknown): error is ExpressionError =>
  error instanceof ExpressionError || error instanceof ExpressionExtensionError;
PLAIN ENGLISH

Detect whether we are running in the browser or on the server — expressions evaluate in both places, which is how the editor shows you a live preview before running.

A helper that checks: is this error a JavaScript syntax error (like a typo in your expression)?

It checks both: is the error an actual SyntaxError object, or does it have the name 'SyntaxError' — because different environments create errors differently.

A different helper: is this error specifically an expression evaluation error (like referencing a field that does not exist)?

⚠️
Dual-Environment Design

This dual-environment design is why you can preview expression results in the editor before running the workflow — the same evaluator runs in your browser for the preview and on the server for the real execution.

When Things Break

Every workflow fails eventually. Here is how to read the evidence, freeze the moment, and never miss a failure again.

Every workflow run is recorded. Every one.

When something breaks, you do not need to reproduce it — n8n already captured it. The execution log shows you every run: what triggered it, what data flowed through each node, where it failed, and what the error was.

Think of it like a flight data recorder — the black box that captures every moment of a flight so investigators can reconstruct exactly what happened. n8n's execution log is your black box: it records the input, output, and status of every node in every run, letting you replay the exact moment things went wrong.

1
Go to the Executions panel

You will see a list of every recent run with status — green checkmark for success, red X for error, and the exact timestamp of each.

2
Click the failed execution

The workflow canvas opens, showing which node failed highlighted in red — you instantly see where in the chain the problem occurred.

3
Click the failed node

See the exact input items it received and the full error message — including the stack trace from the external API if one was thrown.

4
Compare with a successful execution

Open a previous working execution and compare the node's input — find what was different in the data that caused this run to fail.

5
Fix, pin, and re-run

Fix the issue, pin the problem data to reproduce it instantly, and re-run to verify your fix works.

Pinned data: stop time and test with real inputs.

Imagine being able to pause a movie at the exact frame where something went wrong, then re-run every scene after that frame with the same frozen input. That is what pinned data does in n8n — you freeze the upstream output and iterate on the downstream logic at full speed.

📌
Freeze a node's output

Right-click any node and pin its output. It will always return that saved data instead of making a live call — perfect for testing expensive or hard-to-reproduce scenarios.

🔁
Re-run downstream safely

With the trigger and early nodes pinned, you can click Execute Workflow and only the unpinned nodes actually run — dramatically faster iteration.

What happens when a workflow crashes?

By default, a failed workflow execution just silently logs the error. But you can attach an Error Workflow — a separate workflow that automatically runs when the first one fails and can alert your team immediately.

CODE 

export class WorkflowExecutionService {
  constructor(
    private readonly logger: Logger,
    private readonly errorReporter: ErrorReporter,
    private readonly failedRunFactory: FailedRunFactory,
    private readonly eventService: EventService,
  ) {}
PLAIN ENGLISH

This service handles running workflows — it needs several helpers injected at construction time.

Logger: records what happens for debugging — writes lines to the application log file.

ErrorReporter: sends errors to monitoring systems (like Sentry) so your engineering team sees production problems.

FailedRunFactory: builds the data object for a failed execution record — formats all the error details into the structure saved to the database.

EventService: publishes events so other parts of the system know what happened — this is what triggers the Error Workflow to fire.

📡
One Event, Many Reactions

The EventService publish-subscribe pattern means when a workflow fails, multiple systems can react: the Error Workflow fires, the execution log updates, monitoring alerts trigger — all from one event.
